# Phase 3d v1.1 — Real DMO shapes (live-probe correction)

> **Status:** Drafted 2026-05-27. Supersedes the cross-DMO field assumptions in `docs/superpowers/specs/2026-05-27-phase-3d-cross-dmo-segments-design.md`.
> **Trigger:** Phase 3d Task 7 step 7.1 (live probe) failed with `ENTITY_SAVE_ERROR: column ssot__PersonLifeEvent__dlm.EventDate__c does not exist`. Subsequent describe + DC SQL probes against `jdo-uqj0jr` revealed the original v1.0 spec assumed a CRM-style flat-FK schema, but the org's standard `ssot__*` DMOs use the SSOT-canonical relationship model (Individual-mediated indirection for person scopes, direct AccountId for business scopes, type-coded enums rather than sub-type fields).
> **Tasks 1–6 of v1.0 still hold** — the DSL extensions (`related_to`, `relative_date_*`), the probe infra, the recreate loader, and the CLI wiring are all correct and committed. Only Task 7 (the YAML rewrite + live execute) needs to be redone against the real schema.

---

## 1. Why v1.1

v1.0 §4.1 specified the new `related_to` rule type with a default `via: AccountId__c` and assumed every related DMO would have an `AccountId__c` foreign key. The live probe disproved this for all four target DMOs in the JDO org. The same probe also exposed:

- The `wealth_recent_life_event` segment's date field is `ssot__PersonLifeEventDateTime__c`, not `EventDate__c`.
- `ssot__FinancialAccount__dlm` has no Mortgage / HELOC / SBA / Treasury sub-type field — it has `ssot__FinancialAccountType__c` with the bucket vocabulary `{Deposits, Loans, Investments, Treasury Management, Credit Cards, Merchant Services, ...}`.
- `ssot__CampaignMember__dlm` stores `ssot__CampaignId__c` as a Salesforce Campaign 18-char ID (e.g., `701am00002A9aDaAAJ`), not the YAML-friendly `HYDRATE-CMP-NNN` external-ID string.
- `ssot__CampaignMember__dlm` splits person vs. business membership across two FK columns: `ssot__IndividualId__c` (B2C) and `ssot__BusinessAccountId__c` (B2B).

These are concrete, measurable shapes from live describe + queryv2 calls. v1.1 captures them as the source of truth and revises Task 7 accordingly.

## 2. Real DMO shapes (live evidence)

Captured from live describe + queryv2 in `output/phase3d/dmo_describes.json`. Summarised here:

### 2.1 `ssot__Account__dlm` (PROFILE, 257 fields)

The Account DMO acts as the segment root. Two relevant FKs:

| Account field | Points to |
|---|---|
| `ssot__IndividualId__c` | Individual DMO (`ssot__Individual__dlm.ssot__Id__c`) for person accounts. Populated for ~25,510 person rows. |
| `ssot__Id__c` | Account DMO's own primary key. Also populated as the Salesforce Account ID (`001am00000…`). Used by B2B-rooted DMOs that store a `BusinessAccountId__c`. |

### 2.2 `ssot__FinancialAccount__dlm` (PROFILE, 40 fields, 32,140 rows)

| Field | Notes |
|---|---|
| `CustomerID__c` | **Foreign key to Account.** Holds the Salesforce Account 18-char Id (`001am00000qvjsAAAQ` and similar). Joins to `Account.ssot__Id__c` (= `Account.Id`). |
| `ssot__FinancialAccountType__c` | Bucket type. **Live-confirmed values**: `Deposits` (22170), `Loans` (8099), `Investments` (1573), `Treasury Management` (243), `Credit Cards` (32), `Merchant Services` (3), `Checking` (6), plus a few small-N legacy values. |
| `ssot__Name__c` | Free-text product name (e.g., "Cumulus Money Market"). Useful for further partitioning Loans into Mortgage/HELOC/SBA when the type-bucket is too coarse — but only if the name string contains the sub-type. Verify per-segment before relying on it. |
| `Average_Balance__c`, `Current_Balance__c`, `Interest_Rate__c`, `Maturity_Date__c`, `Open_Date__c` | Useful for follow-on partitioning (e.g., HELOC drawn ratio if the data is loaded that way). |

**v1.0 referenced these non-existent fields:** `AccountId__c`, `FinServ_AccountType_c__c`, `FinServ_DrawnRatio_c__c`. None exist in this DMO.

### 2.3 `ssot__CampaignMember__dlm` (PROFILE, 17 fields, 57,872 rows)

| Field | Notes |
|---|---|
| `ssot__BusinessAccountId__c` | **B2B FK** → `Account.ssot__Id__c`. Populated for business-account memberships. |
| `ssot__IndividualId__c` | **B2C FK** → `Individual.ssot__Id__c` → join to person Account via `Account.ssot__IndividualId__c`. |
| `ssot__CampaignId__c` | 18-char SF Campaign Id. **Mapping captured** for all 10 hydrated campaigns: |

```text

HYDRATE-CMP-002 -> 701am00002A9aDbAAJ
HYDRATE-CMP-003 -> 701am00002A9aDcAAJ

HYDRATE-CMP-005 -> 701am00002A9aDeAAJ
HYDRATE-CMP-006 -> 701am00002A9aDfAAJ
HYDRATE-CMP-007 -> 701am00002A9aDgAAJ

HYDRATE-CMP-009 -> 701am00002A9aDiAAJ
HYDRATE-CMP-010 -> 701am00002A9aDjAAJ
```

**v1.0 referenced these non-existent fields:** `AccountId__c`, `CampaignId__c` (without the `ssot__` prefix), and assumed `CampaignId` could be matched against the literal `HYDRATE-CMP-NNN` string.

### 2.4 `ssot__PersonLifeEvent__dlm` (ENGAGEMENT, 21 fields, 25,287 rows)

| Field | Notes |
|---|---|
| `ssot__IndividualId__c` | **FK** → `Individual.ssot__Id__c` → join to person Account via `Account.ssot__IndividualId__c`. |
| `ssot__PersonLifeEventDateTime__c` | DateTime field. **Date range observed**: `1969-03-15` → `2026-11-16`. (Augment-phase3 forward-dates events for demo realism.) |
| `ssot__PersonLifeEventType__c` | Live-confirmed type vocabulary: `Home`, `Job`, `Baby`, `Graduation`, `Retirement` (top 5), plus small-N values like `Marriage`, `Birth`, `College`, `Relocation`, `Car`, `New Business`. |

**v1.0 referenced these non-existent fields:** `AccountId__c` (used `IndividualId` instead) and `EventDate__c` (the field is named `ssot__PersonLifeEventDateTime__c`).

### 2.5 `ssot__Individual__dlm` (PROFILE, 145 fields)

The Individual DMO is the join hub for person-rooted cross-DMO clauses. Relevant fields:

| Field | Notes |
|---|---|
| `ssot__Id__c` | Primary key. Joined from `Account.ssot__IndividualId__c`. |
| `ssot__PrimaryAccountId__c` | Reverse FK to Account (alternative path). |

For Phase 3d: we do **not** need to traverse Account → Individual → PersonLifeEvent in two hops. The simpler one-hop expression is "Account.ssot__IndividualId__c MATCHES PersonLifeEvent.ssot__IndividualId__c" — both DMOs key on the same Individual Id, so a NestedAttribute can match them directly.

## 3. Revised cross-DMO clauses (per segment)

Each row maps a v1.0 segment intent to its v1.1 expression. The translator's `related_to` rule type from Task 3 is unchanged; only the YAML inputs differ.

### 3.1 `retail_family_with_mortgage` — Retail with at least one Loan-type FinancialAccount

```yaml
rule:
  type: all_of
  rules:
    - type: text_equals
      field: FinServ_ClientCategory_c__c
      value: "Retail"
    - type: related_to
      dmo: ssot__FinancialAccount__dlm
      via: CustomerID__c
      where:
        type: text_equals
        field: ssot__FinancialAccountType__c
        value: "Loans"
```

**Coarsening note:** The org has no Mortgage sub-type on FinancialAccount. v1.1 segments on `Loans` (8,099 rows). A future Phase 3e could add a `text_contains ssot__Name__c "Mortgage"` clause if the loan names include that substring; for v1.1 we accept the coarser bucket.

### 3.2 `retail_heloc_drawn` — Retail with a Loan-type account whose name suggests HELOC

```yaml
rule:
  type: all_of
  rules:
    - type: text_equals
      field: FinServ_ClientCategory_c__c
      value: "Retail"
    - type: related_to
      dmo: ssot__FinancialAccount__dlm
      via: CustomerID__c
      where:
        type: all_of
        rules:
          - type: text_equals
            field: ssot__FinancialAccountType__c
            value: "Loans"
          - type: text_contains
            field: ssot__Name__c
            value: "HELOC"
```

**Coarsening note:** The drawn-ratio clause from v1.0 is dropped — there is no `Drawn_Ratio` field on the live DMO. Implementation will verify the `text_contains "HELOC"` filter actually matches some rows before committing; if zero rows, fall back to the same expression as `retail_family_with_mortgage`.

### 3.3 `smb_with_sba` — SMB with a Loan-type account whose name suggests SBA

```yaml
rule:

      value: "Small Business"
    - type: related_to
```

Same fallback as 3.2 if the name search returns 0.

### 3.4 `commercial_with_treasury` — Commercial with a Treasury Management account

```yaml
      field: FinServ_ClientCategory_c__c
      value: "Commercial Banking"
    - type: related_to

```

**Note:** This one is the cleanest of the lot — the live DMO has a real `Treasury Management` enum value with 243 rows.

### 3.5 `wealth_recent_life_event` — Wealth with PersonLifeEvent in the last 90 days

```yaml
rule:
  type: all_of
  rules:
    - type: text_equals
      field: FinServ_ClientCategory_c__c
      value: "Wealth Management"
    - type: related_to
      dmo: ssot__PersonLifeEvent__dlm
      via: ssot__IndividualId__c
      via_root: ssot__IndividualId__c   # NEW: the Account-side field to join FROM
      where:
        type: relative_date_after_days
        field: ssot__PersonLifeEventDateTime__c
        days: 90
```

**Important new DSL knob:** When the join is via Account.IndividualId (not Account.Id), the YAML rule needs to override the Account-side join field. v1.0 `related_to` defaults `primaryFieldApiName: "Id"` in the emitted JSON. v1.1 introduces an optional `via_root:` key to override this. **Translator change required** (see §4).

### 3.6–3.13 Person-targeted campaign segments (cmp_heloc_refi_outreach, cmp_auto_loan_rate_drop, cmp_premier_checking_onboarding, cmp_wealth_tax_strategy_webinar, cmp_wealth_estate_planning_roundtable, cmp_sba_awareness, cmp_treasury_modernization_brief, cmp_commercial_rm_roundtable, cmp_mobile_banking_adoption)

For person-rooted personas (retail, wealth) and SMB (which our hydration writes as Person Accounts in this org), use the IndividualId path:

```yaml
rule:
  type: all_of
  rules:
    - type: text_equals
      field: FinServ_ClientCategory_c__c
      value: "<persona>"
    - type: related_to
      dmo: ssot__CampaignMember__dlm
      via: ssot__IndividualId__c
      via_root: ssot__IndividualId__c
      where:
        type: text_equals
        field: ssot__CampaignId__c
        value: "<resolved 18-char Campaign Id from 2.3 mapping>"
```

For commercial / treasury campaigns where the membership is logged on the business account: use the BusinessAccountId path:

```yaml
- type: related_to
  dmo: ssot__CampaignMember__dlm
  via: ssot__BusinessAccountId__c
  # via_root defaults to "Id" / ssot__Id__c — Account's primary key
  where:
    type: text_equals
    field: ssot__CampaignId__c
    value: "<resolved 18-char Campaign Id>"
```

**Operator action required:** Implementation chooses one of these two patterns per campaign segment, based on whether the campaign was loaded with person or business members. The mapping for v1.1 (initial pass):

## 4. Required code changes for v1.1

Only **one** change to the existing translator:

**Add `via_root:` support to the `related_to` rule branch** in `customer_hydration/phase5/segments.py` (`_translate_rule`):

```python
    if rule_type == "related_to":
        related_dmo = rule.get("dmo")
        if not related_dmo:
            raise ValueError(...)
        via = rule.get("via", "AccountId__c")
        via_root = rule.get("via_root", "Id")   # NEW: default unchanged for back-compat
        where = rule.get("where")
        ...
        return {
            "type": "NestedAttribute",
            "primaryObjectApiName": target_dmo,
            "primaryFieldApiName": via_root,    # NEW: was hardcoded "Id"
            "relatedObjectApiName": related_dmo,
            "relatedFieldApiName": via,
            "filter": inner,
        }
```

**Test additions** to `tests/test_segments_translator_related_to.py`:

```python
def test_related_to_via_root_overrides_primary_field(tmp_path: Path):
    yaml_path = _write(tmp_path, """\
segments:
  via_root_test:
    name: "x"
    description: "x"
    persona: wealth
    publish_schedule: daily
    target_dmo: ssot__Account__dlm
    rule:
      type: related_to
      dmo: ssot__PersonLifeEvent__dlm
      via: ssot__IndividualId__c
      via_root: ssot__IndividualId__c
      where:
        type: text_has_value
        field: ssot__PersonLifeEventDateTime__c
""")
    defs = load_segment_definitions(yaml_path)
    user = defs[0].include_criteria["filters"][1]
    assert user["primaryFieldApiName"] == "ssot__IndividualId__c"
    assert user["relatedFieldApiName"] == "ssot__IndividualId__c"
```

**Default-via change:** v1.0 defaulted `via: AccountId__c`. None of v1.1's segments use that default — every cross-DMO clause supplies an explicit `via`. We can keep the default as `AccountId__c` for back-compat (the four existing tests in `test_segments_translator_related_to.py` rely on it) but the YAML rewrite from §3 always names `via` explicitly.
