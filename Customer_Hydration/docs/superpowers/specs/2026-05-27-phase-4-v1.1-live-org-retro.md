# Phase 4 v1.1 — Live-org compatibility retro

> **Status:** Closed-out. All hotfixes merged to main as of `4cbbb9f`.
> **Origin:** Phase 4 (`docs/superpowers/specs/2026-05-26-phase-4-account-backfill-design.md`) shipped as four plans (4a → 4d). v1.0 passed all 786 unit tests against fixtures, but the first live-org runs against `jdo-uqj0jr` exposed a series of **schema-drift and platform-policy** mismatches that fixtures cannot catch. This doc captures the four hotfix waves that closed those gaps so the Phase 4 backfill behaves correctly in any FSC org with non-default picklists, restricted writability, or stream policies that block manual REST refresh.

---

## 1. Why a v1.1 was needed

v1.0's failure mode was uniform: **"works against fixtures, fails against the org."** The deriver registry, coverage rules, and CSV/Bulk pipeline were correct; the gaps were all at the boundary where synthetic data meets the org's actual schema.

Three classes of drift surfaced:

1. **Field-level drift** — formula/rollup/managed-pkg fields that fixtures treat as updatable but the platform rejects on update.
2. **Picklist-value drift** — fixtures use the spec's vocabulary (Tier=Diamond/Platinum/...), the org uses a custom restricted vocabulary (Tier=A/B/C).
3. **Range/format drift** — fixtures don't enforce the org's declared `precision`/`scale` or its calendar quirks (Feb 29 + non-leap year).

Plus one orchestration gap (DC stream refresh policy classification).

---

## 2. Hotfix waves

Each wave is one or more commits clustered around a coherent failure class. Commits cited inline.

### Wave 1 — Writability preflight (`f4282b8`)

**Symptom:** First live run against `jdo-uqj0jr` returned 100% bulk failures with `INVALID_FIELD_FOR_INSERT_UPDATE` on `FinServ__BillingAddress__pc`, `FinServ__LengthOfRelationship__c`, `FinServ__ShippingAddress__pc`, `JigsawCompanyId`. These are formula/rollup/managed-package read-only fields that fixtures never flagged because nothing in unit tests calls `Account.describe()`.

**Fix:** Added `find_unwritable_fields()` (in `customer_hydration/backfill/preflight.py`) that runs `sf sobject describe Account` once at startup, walks `fields[]`, and returns any field whose `updateable=false` for fields in the deriver registry's owned set. Orchestrator masks these out of every row's `delta` before CSV write.

**Why describe is the right boundary:** It's authoritative for the org's current state, including managed-package upgrades that may flip writability between releases. A static blocklist would rot.

**Failure-mode gate:** When describe itself fails (network/auth), the preflight returns an empty set rather than raising — better to attempt the upsert and let Bulk API report row-level rejections than to over-drop and lose the run entirely.

### Wave 2 — Picklist drift preflight + value translator + length-tolerant `weighted_pick` (`1aac02d`)

**Symptom A — restricted picklist rejection:** `INVALID_OR_NULL_FOR_RESTRICTED_PICKLIST: Tier__c — value "Bronze" is not in the allowed list` (org accepts only `A`, `B`, `C`).

**Symptom B — sample-size mismatch:** When the picklist filter cut a 4-value list down to 1, `weighted_pick(values, weights)` raised `values has 1 items but weights has 3`.

**Symptom C — semantic drift:** Some org values are *spelled* differently from the spec's vocabulary (`ServiceModel`: spec says `Premier`, org says `Tier 1`). A simple drop-filter would lose the distribution shape entirely.

**Fix (three parts):**

1. **`find_picklist_drift()`** — runs describe per restricted picklist field, intersects YAML values with org-accepted values, returns drift report. `filter_picklist_yaml_to_org()` produces a filtered YAML; `install_picklist_overrides()` swaps it into `_PICKLIST_OVERRIDE` in `_helpers.py` so derivers' `weighted_pick(values_for_field("Tier__c"))` calls only see org-accepted values.
2. **Length-tolerant `weighted_pick`** — when `len(values) < len(weights)`, truncates weights; when longer, pads with the trailing weight value. Preserves shape when filtering removes options.
3. **Value translator** — `customer_hydration/backfill/value_translator.py` + `config/account_value_translator.yaml` — runs *after* derivation, replaces well-known semantic equivalents (Diamond → A, Premier → Tier 1). YAML is empty by default; only kept fields are explicitly mapped, no implicit re-spelling.

**Why three layers:** Filter-first avoids picking forbidden values at all; translate-second preserves the spec's intent for known synonyms; tolerant `weighted_pick` is a defense-in-depth so any *unknown* future drift just narrows the distribution rather than crashing.

### Wave 3 — Numeric-range preflight + persona-prefix list mode + External_ID requirement flip (`a875d00`)

**Symptom A — number out of range:** D&B Failure Score deriver emits 1001-1610, but the org's `DNB_Failure_Score__c` was created with `precision=3, scale=0` → max `999`. Bulk rejected ~30% of B2B rows.

**Symptom B — `--persona retail` returned 0 records:** `PERSONA_PREFIX_MAP` had `retail → "HYDRATE-RTL-"`, but `jdo-uqj0jr`'s External_IDs use `HYDRATE-RT-` (no L). Singular-string mapping couldn't accept both.

**Symptom C — synthetic IDs failed on org-seed records:** `BACKFILL-<Id>` synthetic External_IDs caused Bulk to treat rows as creates rather than updates, which fails for MDM seed records (can't set `__pc` fields on non-person account creates).

**Fix:**

1. **`numeric_field_constraints()` + `value_exceeds_field_range()`** — describe walks numeric fields, computes `max = 10^(precision-scale) - 10^(-scale) - epsilon`. Per-row check drops any value exceeding the constraint and increments a counter. Manifest includes `numeric_drops_count` for visibility.
2. **`PERSONA_PREFIX_MAP` is now `dict[str, list[str]]`** — `retail → ["HYDRATE-RTL-", "HYDRATE-RT-"]`. Query builder generates `(External_ID__c LIKE '...%' OR External_ID__c LIKE '...%')`. Adding new orgs is one-line config.
3. **`require_external_id` default flipped from `False` to `True`** — records without External_ID are now skipped rather than synth-stamped. CLI flag renamed from `--require-external-id` to `--allow-missing-external-id` to make the dangerous opt-in explicit.

**Why default-skip:** synthetic stamping looks safe for a fixture but creates unrecoverable Bulk state in real orgs (creates that should have been updates, then re-runs that re-create as new rows). Skip-by-default + explicit opt-in matches the "safe by default" convention in [[feedback_executing_actions_with_care]] reasoning.

### Wave 4 — Cleanup wave: leap year, gitignore, picklist YAML refresh (`ecaa094`)

**Symptom A — leap-year crash:** `date.replace(year=2007)` raised `ValueError: day is out of range for month` when the synthetic birthdate was Feb 29 and the demographics deriver tried to derive `EmployedSince` by setting `year = birth_year + 18` (a non-leap year). One unlucky persona seed broke ~0.003% of rows.

**Symptom B — uncommitted run dirs:** `output/backfill-accounts-*/` was being staged by accident.

**Symptom C — stale picklist YAML:** the in-repo `config/backfill_picklists.yaml` still had spec-vocabulary values (Tier=Diamond/Platinum/Gold/Silver/Bronze). The picklist drift preflight worked around it at runtime, but the YAML drift confused readers.

**Fix:**

1. **Leap-year clamp:** in `demographics.py`, wrap `birthdate.replace(year=target_year)` in `try/except ValueError` and fall back to `replace(year=target_year, day=28)`. Localized at one site; no calendar library dependency.
2. **`.gitignore`:** added `output/backfill-accounts-*/` and `output/augment-phase3-*/`. Audit dirs still keep `REPORT.md` per existing rules.
3. **YAML refresh:** edited `config/backfill_picklists.yaml` to match `jdo-uqj0jr`'s actual vocabulary. The runtime override layer remains as defense-in-depth for any future divergence.

### Wave 5 — DC stream refresh policy classifier (`78deb80`)

**Symptom:** After all bulk fixes landed, the live run completed successfully but `dc_refresh.py` returned `Failed` (rc=2) on streams whose `refreshConfig.refreshMode = UPSERT`. The REST `actions/run` endpoint returns HTTP 412 with messages like `"Manual run not allowed for non-FULL_REFRESH refresh policy"`. v1.0 classified this as a hard failure.

**Fix:** Broadened the classifier in `refresh_account_stream()` to recognize keywords `policy`, `full_refresh`, `not allowed`, `non-interactive` in the error string and return `PolicySkipped` (not `Failed`). The orchestrator already had this status path wired but only for one specific message shape.

**Why this matters:** PolicySkipped is *expected* for any FSC-source stream in the JDO orgs — they're all UPSERT-mode. The right manual-trigger path for those is the `dc-stream-full-refresh-via-ui` skill (Lightning UI). Misclassifying as Failed would have made every Phase 4 run end with `rc=2` and obscured genuine bulk-load failures.

## 4. Open seams (NOT fixed in v1.1)

These are known soft spots worth tracking:

1. **`hydrate.py dc-status` doesn't surface `PolicySkipped` from the manifest.** It reads the stream's last-run status from REST, which shows the UI-driven Full Refresh result but doesn't echo the orchestrator's classification. Low impact (the manifest is the source of truth for the run), but a one-line `dc-status --include-manifest <run-dir>` toggle would make the operator UX cleaner.

2. **No describe-cache.** Each preflight calls `sf sobject describe Account` independently. For Phase 4 it's ~3 calls per run. If we add similar pipelines for other objects, an in-process describe cache (process-local, no persistence) would cut latency.

3. **Value translator YAML is hand-curated.** Adding a new org variant requires reading the live picklist values and writing the translation. A `hydrate.py suggest-translations --target-org <alias>` helper that diffs YAML vs. live and proposes mappings would shrink onboarding to seconds.

4. **No structured retry.** Bulk failures during partial-success runs are reported but not retried. A `--retry-failed` flag that reads the failed-records CSV and re-submits only those rows would help when the failure is transient (e.g., row-locking against a parallel pipeline).

These don't block production use; they're documented here so future work has a starting point.

## 5. Lessons for the spec/plan workflow

- **Live-org smoke tests are non-negotiable for cross-org pipelines.** Fixtures are necessary but never sufficient. The Plan 4d "live smoke gated by `RUN_LIVE_TESTS`" pattern was right; we should bake it into Plan 5+ as a checklist item.
- **Schema-drift preflights are reusable.** Writability + picklist-drift + numeric-range patterns belong in a shared `customer_hydration/backfill/preflight.py` library, which we now have. Future deriver pipelines (Contact, Lead, Opportunity) can lift them wholesale.
- **Default-skip beats default-stamp.** Whenever a deriver pipeline can write to live records, the safe default is to skip ambiguity rather than synthesize a placeholder. Plan 4d originally had the inverse default; the live run made it clear why that's wrong.
