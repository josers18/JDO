## 1. The actual v62 cross-DMO shape

Captured live from `/services/data/v62.0/ssot/segments` payload of `Recent_Transactions` (a UI-built segment that uses the same Account-rooted cross-DMO pattern Phase 3d needs). Persisted to `output/phase3d/recent_transactions_includecriteria.json`.

The `Recent_Transactions` filter that joins `ssot__Account__dlm` to `ssot__AccountContact__dlm` (and counts matching rows ≥ 1) is shaped like this:

```json
{
  "type": "NumberAggregation",
  "aggregateFunction": "count",
  "containerObjectApiName": "ssot__AccountContact__dlm",
  "path": [
    [
      {"objectApiName": "ssot__Account__dlm", "fieldApiName": "ssot__Id__c"},
      {"objectApiName": "ssot__AccountContact__dlm", "fieldApiName": "ssot__AccountId__c"}
    ]
  ],
  "joinPath": [
    [
      {"objectApiName": "ssot__Account__dlm", "fieldApiName": "ssot__Id__c"},
      {"objectApiName": "ssot__AccountContact__dlm", "fieldApiName": "ssot__AccountId__c"}
    ]
  ],
  "filter": {
    "type": "TextComparison",
    "path": null,
    "joinPath": null,
    "subject": {"objectApiName": "ssot__AccountContact__dlm", "fieldApiName": "ssot__Title__c"},
    "selfReference": false,
    "operator": "in",
    "subjectFieldDataType": "TEXT",
    "subjectFieldBusinessType": "TEXT",
    "subjectFieldSourceType": "RELATED",
    "values": ["Physician", "Dr", "Doctor"]
  },
  "comparison": {
    "type": "NumberComparison",
    "path": null,
    "joinPath": null,
    "subject": {"objectApiName": "ssot__AccountContact__dlm", "fieldApiName": "ssot__Id__c"},
    "selfReference": false,
    "operator": "greater than or equal",
    "subjectFieldDataType": "TEXT",
    "subjectFieldBusinessType": "TEXT",
    "subjectFieldSourceType": null,
    "value": 1
  },
  "hierarchySelected": false,
  "hierarchicalPathList": null,
  "innerAggregationEnabled": false,
  "innerAggregationSubject": null,
  "outerAggregationFunction": null,
  "outerComparison": null
}
```

Semantics: "the count of rows in `ssot__AccountContact__dlm` joined back to `ssot__Account__dlm.ssot__Id__c` via `ssot__AccountId__c` and where `ssot__Title__c IN [Physician, Dr, Doctor]` is ≥ 1." Functionally a SQL `EXISTS`.

### 1.1 Required fields

| Field | Value | Notes |
|---|---|---|
| `type` | `"NumberAggregation"` | The outer type. |
| `aggregateFunction` | `"count"` | Always count for "exists"-style filters. |
| `containerObjectApiName` | The related DMO | Same as the inner filter's subject DMO. |
| `path[0][0]` | `{root_dmo, root_field}` | Account-side join field. Phase 3d v1.2 default: `{"objectApiName": <target_dmo>, "fieldApiName": "ssot__Id__c"}`. Override via `via_root`. |
| `path[0][1]` | `{related_dmo, fk_field}` | Related-DMO side. Comes from YAML `via:`. |
| `joinPath` | Mirrors `path` | Required separately by the API; same shape. |
| `filter` | The translated `where:` rule | With added `subjectFieldDataType` / `subjectFieldBusinessType` / `subjectFieldSourceType: "RELATED"` annotations for text/number comparisons. |
| `comparison` | Pinned shape | `NumberComparison` on `<related_dmo>.ssot__Id__c >= 1`. Identical for every `related_to` clause. |
| Pinned defaults | `hierarchySelected: false`, `hierarchicalPathList: null`, `innerAggregationEnabled: false`, `innerAggregationSubject: null`, `outerAggregationFunction: null`, `outerComparison: null` | Always present; always these literal values for our use case. |

### 1.2 Field-type annotations on the inner `filter`

The live UI-built filters carry per-comparison metadata: `subjectFieldDataType`, `subjectFieldBusinessType`, `subjectFieldSourceType`, `selfReference`, `path: null`, `joinPath: null`. For Phase 3d's translator:

- `subjectFieldDataType` / `subjectFieldBusinessType`: derive from the DC type. v1.2 default rules:
  - `text_*` → both `"TEXT"`
  - `number_*` → both `"NUMBER"`
  - `date_*` / `relative_date_*` → both `"DATE"`
- `subjectFieldSourceType`: `"RELATED"` when the subject DMO is the related DMO (which it always is inside a `related_to` clause). For the outer Account-rooted filters (`text_equals FinServ_ClientCategory_c__c "Retail"`), this annotation is **not** required by the existing Phase 2 segments and shouldn't be added there.
- `selfReference`: `false` (always for Phase 3d).
- `path: null`, `joinPath: null` on the inner filter: required by some API paths; safe to always emit on inner comparisons inside `related_to`.

### 2.2 New `_annotate_inner_filter(node, container_dmo)` helper

Walks a translated comparison node and adds the `subjectFieldDataType`/`subjectFieldBusinessType`/`subjectFieldSourceType` annotations + nullable `path`/`joinPath` keys when the subject DMO equals `container_dmo`. Recurses into `LogicalComparison` filters.

```python
_DSL_TYPE_ANNOTATIONS = {
    "TextComparison":            ("TEXT", "TEXT"),
    "NumberComparison":          ("NUMBER", "NUMBER"),
    "DateComparison":            ("DATE", "DATE"),
    "ExactlyRelativeDateComparison": ("DATE", "DATE"),
}


def _annotate_inner_filter(node: dict[str, Any], container_dmo: str) -> dict[str, Any]:
    """Add live-API metadata that NumberAggregation's nested filters require.

    The DC v62 segment POST endpoint expects every comparison nested inside
    a NumberAggregation to carry subjectFieldDataType / subjectFieldBusinessType
    / subjectFieldSourceType / selfReference / path:null / joinPath:null.
    The translator builds bare comparisons (TextComparison, NumberComparison,
    etc.) and this helper layers the annotations on without changing semantics.
    """
    if not isinstance(node, dict):
        return node
    out = dict(node)  # shallow-copy so we don't mutate caller state
    node_type = out.get("type")

    # Recurse into compound rules first.
    if node_type == "LogicalComparison":
        out["filters"] = [_annotate_inner_filter(f, container_dmo) for f in out.get("filters", [])]
        return out

    # Atomic comparisons: add the annotations.
    annotations = _DSL_TYPE_ANNOTATIONS.get(node_type)
    if annotations is None:
        return out  # unknown type — leave untouched

    data_type, business_type = annotations
    out.setdefault("subjectFieldDataType", data_type)
    out.setdefault("subjectFieldBusinessType", business_type)
    subject_dmo = out.get("subject", {}).get("objectApiName")
    if subject_dmo == container_dmo:
        out.setdefault("subjectFieldSourceType", "RELATED")
    else:
        out.setdefault("subjectFieldSourceType", None)
    out.setdefault("selfReference", False)
    out.setdefault("path", None)
    out.setdefault("joinPath", None)
    return out
```
