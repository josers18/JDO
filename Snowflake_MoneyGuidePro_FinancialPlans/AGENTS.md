# AGENTS.md — Snowflake_MoneyGuidePro_FinancialPlans

Synthetic MoneyGuidePro / eMoney / NaviPlan-style financial-plan dataset for the Cumulus FSC demo. One of 13. **Account-scoped with monthly cadence — the smallest Cumulus audience by 2.9× (Wealth Management only, ~3,920 anchors) AND the first dataset whose NULL semantics are gated by a non-Boolean enum.**

> **v1.x multi-org-additive (Phase A, 2026-05-29 commit `c9119d32`).** Table now leads with `ORG_ID VARCHAR(18) NOT NULL DEFAULT 'JDO'` as the first column; PK promoted from `(ACCOUNT_ID, PROFILE_MONTH)` to `(ORG_ID, ACCOUNT_ID, PROFILE_MONTH)`. SP row factory stamps `"ORG_ID": anchor.get("ORG_ID", "JDO")` as the first key; MERGE source SELECT, ON, INSERT lists all lead with ORG_ID; UPDATE SET deliberately skips ORG_ID (PK-component, immutable). The `num_cycles` parameter on `main()` is unchanged. Backward-compatible — JDO loaders continue working unchanged via DEFAULT. Multi-org rollout runbook: `Snowflake_Cumulus_Common/docs/ROLLOUT.md`.

## Boundaries
- Owns: `FINS.PUBLIC.MGP_FINANCIAL_PLANS`, `SP_GENERATE_MGP_FINANCIAL_PLANS`, `TASK_MONTHLY_MGP_FINANCIAL_PLANS`, and the DC Data Stream / DLO / DMO that federates this table.
- Does NOT own: `V_ACCOUNT_ANCHORS`, `MASTER_ACCOUNTS`, the seed/coverage helpers — see `Snowflake_Cumulus_Common`.
- Does NOT own any outbound Snowflake share. DC reads through via the existing "Snowflake (Federate / Zero Copy)" connector.
- **Account-scoped** — rows are keyed by composite PK `(ACCOUNT_ID, PROFILE_MONTH)`, with FK `ACCOUNT_ID` to `ssot__Account__dlm`. **1:1 — one row per distinct Wealth Management account per month.**

## Conventions
- The SP uses `cumulus_common.seed_for(...)` for determinism with salt `"mgp"` (**month-bucketed**, single-salt — no year-stable streams; back to Plan 6's simpler shape after Plan 7's three salts).
- **One seed stream per anchor per run** — `seed_for(account_id, "mgp", month_start)` where `month_start = run_ts.replace(day=1, hour=0, minute=0, second=0, microsecond=0)`. Drives PLAN_STATUS, PLAN_LAST_UPDATED_DATE, RETIREMENT_TARGET_AGE, MONTHLY_INCOME_TARGET_USD, TOTAL_GOAL_AMOUNT_USD, GOAL_COUNT, MONTE_CARLO_SUCCESS_PCT, RECOMMENDED_ASSET_ALLOCATION, LAST_REVIEW_DATE, NEXT_REVIEW_DATE, ADVISOR_NOTES_FLAG. Mid-month re-runs are byte-identical because `month_start` truncates day + time.
- The SP uses `cumulus_common.assert_coverage(session, expected_sql, actual_sql)` for the standard 1:1 audience-vs-actual coverage check (every distinct Wealth Management anchor must produce exactly one row per month).
- The MERGE replaces on composite PK `(ACCOUNT_ID, PROFILE_MONTH)`. Re-runs same calendar month are idempotent (byte-identical). Re-runs in a later month INSERT new rows for that month.
- Audience SQL is `SELECT DISTINCT * FROM V_ACCOUNT_ANCHORS WHERE CLIENT_CATEGORY = 'Wealth Management'`. 3,920 distinct accounts (probed 2026-05-28). Both BIRTHDATE and ANNUAL_INCOME 100% populated for this cohort — no NULL fallback needed when reading them.
- `accounts_processed` in `TASK_EXECUTION_LOG` equals `len(audience)` — the **distinct customer count** AND the row count (1:1 emit per anchor per month).

## Tests
- L1 (pytest): determinism in the same calendar month (byte-identical at 03:00 vs 23:30, day-1 vs day-15), input-shape validation (ACCOUNT_ID non-empty), boring case (age 50, $300K Wealth anchor → Active most likely, Moderate or Moderate Aggressive allocation, both review dates populated), **per-anchor age-glide invariants** (every age <35 anchor → Aggressive or Moderate Aggressive; every age ≥70 anchor → Moderate Conservative or Conservative), **per-anchor income-floor invariants** (every anchor's MONTHLY_INCOME_TARGET in [income×0.70/12, income×0.90/12]), **NULL-semantics invariants** rolled over 6+ months for diverse PLAN_STATUS sampling (Draft → LAST_REVIEW_DATE NULL; Stale → NEXT_REVIEW_DATE NULL; Active → both populated), **date-coherence invariants** (PLAN_LAST_UPDATED_DATE ≤ today; LAST_REVIEW_DATE if populated ≤ today; NEXT_REVIEW_DATE if populated > today), **range invariants** (RETIREMENT_TARGET_AGE in [55,80], MONTHLY_INCOME_TARGET in [10K,200K], TOTAL_GOAL_AMOUNT in [500K,50M], GOAL_COUNT in [1,6], MONTE_CARLO_SUCCESS_PCT in [30.0,99.0]), schema contract.
- L2 (`tests/integration/`): deploys SP into a fixture-backed schema with `V_ACCOUNT_ANCHORS_FIXTURE` (4 Wealth Management + 10 non-Wealth — audience filter excludes the 10); asserts coverage (`COUNT(DISTINCT ACCOUNT_ID) = 4`), idempotency on same-month re-run (ROWS_INSERTED=0), all 14 columns populated correctly, NULL-semantics invariants enforced (0 Draft rows with LAST_REVIEW_DATE; 0 Stale rows with NEXT_REVIEW_DATE; 0 Active rows missing either), at least 1 of the 3 PLAN_STATUS values present.
- L3 (manual smoke, post-deploy): one CALL against jdo-uqj0jr; row count ≈ 3,920 (distinct Wealth Management accounts); 5-row sample plausibility (mix of Active / Draft / Stale); PLAN_STATUS distribution (~80% Active, ~12% Draft, ~8% Stale within ±1 pp); NULL-semantics post-load checks (0 Draft rows with LAST_REVIEW_DATE; 0 Stale rows with NEXT_REVIEW_DATE; 0 Active rows missing either); date-coherence post-load checks (0 future-dated PLAN_LAST_UPDATED, LAST_REVIEW; 0 non-future NEXT_REVIEW); range invariants (RETIREMENT_TARGET_AGE in [55,80]; MONTHLY_INCOME_TARGET_USD in [10K,200K]; etc.); asset-allocation distribution heavy in Moderate / Moderate Aggressive (most Wealth anchors are 40-65 age band).

## Gotchas
- **Smallest Cumulus audience.** 3,920 anchors vs Plan 2/3's 11,389 (next-smallest). SP runtime expected <2s. Storage trivial. **L1 fixture has only ~3-5 Wealth anchors at SAMPLE_ANCHORS' 100-anchor scale** — far too few for distributional rate convergence. If `SAMPLE_ANCHORS` happens to surface fewer than 3 Wealth anchors, the L1 cohort-specific tests should `pytest.skip` gracefully (Plan 5/6 pattern).
- **Per-anchor deterministic invariants, not distributional.** Property tests for age→allocation, income→monthly_target, NULL semantics, date coherence are per-anchor or per-row (e.g. "every age <35 anchor has Aggressive or Moderate Aggressive allocation"), NOT mean comparisons or rate convergence. Plans 1-7 all relied on rate-convergence tests over SAMPLE_ANCHORS — Plan 8 is the first dataset where the cohort is too narrow for that to work, so the structural shift to per-anchor invariants is load-bearing.
- **Status-driven NULL semantics.** `PLAN_STATUS='Draft'` → `LAST_REVIEW_DATE` NULL (advisor hasn't reviewed yet). `PLAN_STATUS='Stale'` → `NEXT_REVIEW_DATE` NULL (no review scheduled). `PLAN_STATUS='Active'` → both populated. **First Cumulus dataset where NULL semantics depend on a non-Boolean enum.** Plan 7 had 2 NULLables conditional on a Boolean (ADVERSE_MEDIA_HIT) and a rating tier rollup — Plan 8 has 2 NULLables both gated by the same VARCHAR(20) status enum. The L1 NULL-semantic invariants are the load-bearing tests because the cohort is too small for distributional checks.
- **Status-first computation.** PLAN_STATUS is computed before review-dates so the NULL gating decision is determined upfront. Don't reorder the body — `_review_dates(plan_status, ...)` reads the status to decide which fields stay None.
- **Anchor reads.** `BIRTHDATE` (used to derive `age`) + `ANNUAL_INCOME` — both 100% populated for Wealth Management (probed 2026-05-28). No NULL fallback needed. Income range $200K-$1.9M, median $333K. The cohort is genuinely affluent.
- **Date-coherence invariants** (must hold per row, asserted in L1 and L3):
  - `PLAN_LAST_UPDATED_DATE ≤ run_ts.date()` — no future-dated updates.
  - `LAST_REVIEW_DATE` (if populated) `≤ run_ts.date()` — no future-dated reviews. The `_review_dates` helper clamps a stray future draw back to `run_ts.date() - rng.randint(7, 90)`.
  - `NEXT_REVIEW_DATE` (if populated) `> run_ts.date()` — must be in the future. Edge-of-month runs need careful boundary handling (the helper draws 30-540 days ahead, so `> run_ts.date()` always holds).
- Salt is `"mgp"` (monthly cadence, single-salt). Plan 5 had 1 year-stable salt; Plan 7 had 3 (1 main + 2 year-stable); Plan 8 reverts to **just one** — back to Plan 6's simpler shape because there's no year-stable subfield (PLAN_STATUS, allocation, MC%, review dates are all month-bucketed; vendor case ID isn't part of the schema).
- 1 BOOLEAN column (`ADVISOR_NOTES_FLAG`) — declare as `Boolean` in DC DLO + DMO (Plan 5 finding — defaulting to Text fails the Boolean parse). Snowflake stores BOOLEAN natively; the MERGE source SELECT casts via `ADVISOR_NOTES_FLAG::BOOLEAN` to be explicit (write_pandas could mis-type on an empty target).
- 2 NULLable date columns (`LAST_REVIEW_DATE`, `NEXT_REVIEW_DATE`) — both gated by `PLAN_STATUS`. The DDL marks both as NULL; the MERGE source SELECT must preserve `None` → `NULL`.
- The `write_pandas(auto_create_table=True)` mis-types `datetime64[ns]` as `NUMBER(38,0)`. The MERGE source SELECT casts back via `TO_TIMESTAMP_NTZ(GENERATED_AT::NUMBER / 1000000000)` — see template Task 4 §_merge.
- 14 columns total: 12 NOT NULL + 2 NULLable (LAST_REVIEW_DATE, NEXT_REVIEW_DATE).
- L1 conftest provides a synthetic anchor fixture — `SAMPLE_ANCHORS` from Cumulus_Common with `in_audience_anchors = [a for a in all_anchors if a["CLIENT_CATEGORY"] == "Wealth Management"]`.
- The DLO → DMO field mapping must be completed in DC Setup UI for fully-custom DMOs (the API endpoint returns 500). See the recipe at `../Snowflake_Claritas_Demographics/docs/dc-setup-recipe.md` from Plan 1 T7.
- Snowflake DATE columns auto-discover as `MM/dd/yyyy` in the DC data-stream POST body. Use that format for `PROFILE_MONTH`, `PLAN_LAST_UPDATED_DATE`, `LAST_REVIEW_DATE`, `NEXT_REVIEW_DATE`'s `sourceFields` entries, NOT `yyyy-MM-dd`.
- DC mapping descriptions must be ≤521 chars (Plan 6 finding). Trim the DMO description if drafting against the rowspec verbatim.
- `accounts_processed = len(audience) = row count` (1:1 emit per anchor per month) — same shape as Plans 1-3, 5, 7.
- Cadence is **monthly** (`'USING CRON 0 7 1 * * UTC'` — first of the month at 07:00 UTC). Matches Plans 1-3, 6.
- **Volume.** ~3,920 rows/month. Re-runs same month MERGE-replace in place; live storage grows by ~3,920 rows per month.
