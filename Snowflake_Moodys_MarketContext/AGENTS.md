# AGENTS.md — Snowflake_Moodys_MarketContext

Synthetic Moody's Investors Service / Moody's Analytics-style credit-rating + market-context dataset for the Cumulus FSC demo. **Plan 13 is the FINAL Cumulus plan in the rollout (13 of 13).** **Instrument-scoped (NOT account-scoped) with daily cadence — the second non-account-scoped Cumulus plan after Plan 4 (Esri / branch-scoped) AND the second daily-cadence plan after Plan 7 (WorldCheck AML).** Most-divergent instantiation of the dataset template (4 structural deviations from the Plan 8 baseline).

## Boundaries
- Owns: `FINS.PUBLIC.MOODYS_MARKET_CONTEXT`, `SP_GENERATE_MOODYS_MARKET_CONTEXT`, `TASK_DAILY_MOODYS_MARKET_CONTEXT`, and the DC Data Stream / DLO / DMO that federates this table.
- Does NOT own: `INSTRUMENT_UNIVERSE`, the seed/coverage helpers — see `Snowflake_Cumulus_Common` (seed helpers) and the existing trades pipeline (which populates `INSTRUMENT_UNIVERSE`).
- Does NOT own any outbound Snowflake share. DC reads through via the existing "Snowflake (Federate / Zero Copy)" connector.
- **Instrument-scoped** — rows are keyed by composite PK `(TICKER, PROFILE_DATE)`. **There is NO `ACCOUNT_ID` column.** The DMO is not joinable to `ssot__Account__dlm`; `TICKER` is the canonical join key for downstream queries, and account ↔ instrument joins go through the existing trades-pipeline `(account, ticker, position)` graph (the same shape as Plan 4 / Esri being branch-scoped, not account-scoped).

## Conventions
- The SP uses `cumulus_common.seed_for(...)` for determinism with **two salts**:
  - **`moodys`** (day-bucketed via the `_daily_seed(ticker, run_ts)` wrapper) — drives daily-bucketed fields: `DAILY_VOLATILITY_PCT`, `THIRTY_DAY_PRICE_CHANGE_PCT`, `MARKET_CAP_USD` daily noise component.
  - **`moodys_year`** (year-bucketed via `datetime(run_ts.year, 1, 1)`) — drives year-stable editorial fields: `CREDIT_RATING`, `RATING_OUTLOOK`, `OUTLOOK_LAST_CHANGED_DATE`, `FIFTY_TWO_WEEK_HIGH_PRICE`, `FIFTY_TWO_WEEK_LOW_PRICE`, `RATING_AGENCY_FLAG_COUNT`, `LIQUIDITY_TIER`, and the year-stable shares-outstanding base used to derive `MARKET_CAP_USD`.
- **`_daily_seed` wrapper inlined per Plan 7.** Cumulus_Common's `seed_for` is Y-M-only (Plan 0 design choice). For daily cadence, both Plan 7 and Plan 13 inline the same wrapper into their SP module:
  ```python
  def _daily_seed(ticker, run_ts):
      """Day-bucketed seed wrapper. Folds the calendar day into the ticker
      parameter so we get a unique seed per (ticker, calendar day) without
      modifying cumulus_common.seed_for. Identical pattern to Plan 7."""
      day_str = run_ts.strftime("%Y%m%d")
      return seed_for(f"{ticker}|{day_str}", "moodys", run_ts)
  ```
  Plan 7 and Plan 13 are the only two daily-cadence plans; both inline the same wrapper — no need to promote it to Cumulus_Common.
- The SP uses `cumulus_common.assert_coverage(session, expected_sql, actual_sql)` for the standard 1:1 audience-vs-actual coverage check. **Coverage rule:** `COUNT(DISTINCT TICKER) = 2,004` from `INSTRUMENT_UNIVERSE` — every distinct instrument must produce exactly one row per profile day.
- The MERGE replaces on composite PK `(TICKER, PROFILE_DATE)`. Re-runs same calendar day are idempotent (byte-identical). Re-runs on a later day INSERT new rows for that day's profile.
- Audience SQL is `SELECT * FROM FINS.PUBLIC.INSTRUMENT_UNIVERSE` — **no WHERE clause**. The IS_ACTIVE filter from the umbrella spec doesn't apply since the column doesn't exist on `INSTRUMENT_UNIVERSE` (see schema-drift note below).
- `accounts_processed` in `TASK_EXECUTION_LOG` (despite the column's name being account-flavored) equals `len(audience)` — for Plan 13 this is the **distinct instrument count** AND the row count (1:1 emit per instrument per day).

## Tests
- L1 (pytest): determinism on the same calendar day (byte-identical at 03:00 vs 23:30 same UTC date), input-shape validation (TICKER non-empty), boring case (mid-cap Financials TICKER → most likely Baa1/Baa2, Stable outlook, NULL OUTLOOK_LAST_CHANGED_DATE, Tier 3 liquidity), **per-row range invariants** (DAILY_VOLATILITY_PCT in [0,25]; THIRTY_DAY_PRICE_CHANGE_PCT in [-25,25]; MARKET_CAP_USD>0; RATING_AGENCY_FLAG_COUNT in [0,3]), **vocabulary invariants** (CREDIT_RATING in 23-value set; RATING_OUTLOOK in 5-value set; LIQUIDITY_TIER in 4-value set), **year-stable cross-day invariants** (same TICKER on day D and D+30 within the same calendar year → identical CREDIT_RATING, RATING_OUTLOOK, FIFTY_TWO_WEEK_HIGH/LOW_PRICE, RATING_AGENCY_FLAG_COUNT, LIQUIDITY_TIER), **cross-field invariants** (FIFTY_TWO_WEEK_HIGH_PRICE >= FIFTY_TWO_WEEK_LOW_PRICE; FIFTY_TWO_WEEK_HIGH_PRICE >= BASE_PRICE * 0.5; LAST_DATA_REFRESH_AT.date() == PROFILE_DATE), schema contract (output dict keys match the 13 DDL columns).
- L2 (`tests/integration/`): deploys SP into a fixture-backed schema with a synthetic `INSTRUMENT_UNIVERSE_FIXTURE` (10 instruments covering all 7 SECTOR values + a None-SECTOR edge case + a very-low BASE_PRICE edge case); asserts coverage (`COUNT(DISTINCT TICKER) = 10`), idempotency on same-day re-run (ROWS_INSERTED=0), all 13 columns populated, ≥6 rows RATING_OUTLOOK='Stable' (78% target with binomial variance), **day-2 re-run** (run_ts +1 day, same year) — identical year-stable fields (rating, outlook, 52W band, liquidity tier) but DAILY_VOLATILITY_PCT may differ.
- L3 (manual smoke, post-deploy): one CALL against jdo-uqj0jr; row count ≈ 2,004 (distinct instruments); 5-row sample plausibility (mix of ratings / outlooks / sectors); **CREDIT_RATING by SECTOR** matches `_SECTOR_RATING_BIAS` (Financials ≥60% in {A1..Baa3}; Utilities ≥70% in {Aa3..Baa1}; Technology spans Aaa to Caa1); RATING_OUTLOOK distribution (~78% Stable, ~16% Positive+Negative, ~6% Developing+Watch within ±1 pp); **year-stable invariant spot-check** across day-1 and day-2 (same year) for 5 tickers — identical CREDIT_RATING / RATING_OUTLOOK / FIFTY_TWO_WEEK_HIGH/LOW / LIQUIDITY_TIER; cross-field post-load checks (0 rows with HIGH<LOW; 0 rows with MARKET_CAP_USD≤0; 0 rows with LAST_DATA_REFRESH_AT.date() ≠ PROFILE_DATE); LIQUIDITY_TIER mostly Tier 2 / Tier 3, few Tier 1, long Illiquid tail; OUTLOOK_LAST_CHANGED_DATE NULL rate ~62% (80% of the ~78% Stable cohort).

## Gotchas
- **Schema drift caught at draft time.** The umbrella spec §3.3 and the Plan 13 manifest §3.3 describe the audience as `WHERE IS_ACTIVE = TRUE` keyed by `INSTRUMENT_ID`. The actual live `INSTRUMENT_UNIVERSE` schema (probed 2026-05-28) is **`(TICKER VARCHAR(10) PK, INSTRUMENT_NAME VARCHAR(200), SECTOR VARCHAR(50), BASE_PRICE FLOAT)`** — there is **no `INSTRUMENT_ID`** and **no `IS_ACTIVE`** column. Plan 13 keys on `TICKER` and reads the full table unconditionally; same shape of fix as Plan 8's `run_ts.date()` vs `month_start.date()` rowspec drift. **No spec amendment needed** since Plan 13 is the only consumer of `INSTRUMENT_UNIVERSE` from the Cumulus side; the spec text is wrong about column names but the structural intent (instrument-scoped, full-table audience) is correct. Don't reintroduce `WHERE IS_ACTIVE = TRUE` — the column doesn't exist; the SQL would error.
- **Instrument-scoped, NOT account-scoped.** Row factory takes an instrument record `(TICKER, INSTRUMENT_NAME, SECTOR, BASE_PRICE)`, **not** an account anchor. **Don't read BIRTHDATE / ANNUAL_INCOME / CLIENT_CATEGORY / STATE / POSTAL_CODE** — none of those exist on instrument records. The 4 instrument fields above (TICKER, INSTRUMENT_NAME, SECTOR, BASE_PRICE) are the only inputs.
- **No `_anchor_in_audience` predicate.** Every instrument in `INSTRUMENT_UNIVERSE` is in audience (no IS_ACTIVE column, no other filter). The SP's audience-filter step is a no-op compared to Plans 1-3/5/6/8 — the only input-shape check is "TICKER is non-empty". Don't reintroduce a per-row audience predicate.
- **Daily cadence.** `PROFILE_DATE` (DATE column equal to `day_start.date()`), NOT `PROFILE_MONTH`. The seed bucket is day-start `datetime(year, month, day, 0, 0, 0)` via the `_daily_seed` wrapper — mid-day re-runs are byte-identical because the time component is truncated. Plans 1-3, 6, 8 use month-bucketing; Plan 5 uses quarter-bucketing; Plans 7 and 13 are the only daily-cadence plans.
- **Two salts.** `moodys` (daily) for daily-bucketed market signals + `moodys_year` (year-stable) for editorial signals. Same shape as Plan 5 (`corelogic` + `corelogic_year`); applied at daily-cadence here. Keep them separate so the daily volatility/30d-change RNG stream doesn't bleed into the year-stable rating/outlook RNG stream.
- **Hybrid year-stable + daily field model** like Plan 7. Editorial signals (CREDIT_RATING, RATING_OUTLOOK, OUTLOOK_LAST_CHANGED_DATE, FIFTY_TWO_WEEK_HIGH/LOW, RATING_AGENCY_FLAG_COUNT, LIQUIDITY_TIER) are year-stable; market signals (DAILY_VOLATILITY_PCT, THIRTY_DAY_PRICE_CHANGE_PCT, LAST_DATA_REFRESH_AT) are daily-bucketed; MARKET_CAP_USD is hybrid (year-stable shares-outstanding × daily price drift). The year-stable cross-day invariant test is **load-bearing** — it's the structural test for the hybrid model.
- **Snowflake digit-leading column rename.** Snowflake identifiers cannot begin with a digit, so the rowspec's market-data fields are renamed at the DDL level: `52_WEEK_HIGH_PRICE` → `FIFTY_TWO_WEEK_HIGH_PRICE`, `52_WEEK_LOW_PRICE` → `FIFTY_TWO_WEEK_LOW_PRICE`, `30_DAY_PRICE_CHANGE_PCT` → `THIRTY_DAY_PRICE_CHANGE_PCT`. The output dict keys, MERGE source SELECT, and DC field mapping (`fiftyTwoWeekHighPrice__c` etc.) all use the renamed identifiers consistently.
- **0 BOOLEAN, 1 NULLable** (`OUTLOOK_LAST_CHANGED_DATE`) — simplest NULL/Boolean footprint of any Cumulus dataset. **DC Boolean-declaration ceremony is NOT needed** (vs Plans 5/7/8 which had Boolean columns requiring explicit `Boolean` field-type declaration in the DLO + DMO). The single NULLable column needs the standard `None → NULL` preservation in the MERGE source SELECT.
- **OUTLOOK_LAST_CHANGED_DATE NULL semantics.** NULL when `RATING_OUTLOOK = 'Stable'` AND the year-stable RNG draw is in the 80% never-changed bucket — so ~62% of all rows have NULL (80% × 78% Stable). Non-Stable outlooks always have a populated date in the last 0-12 months; Stable outlooks have either NULL (80%) or a stale 6-36-month-ago date (20%).
- **Year-stable seeds folded by `datetime(run_ts.year, 1, 1)`.** The year-stable subseed IDs are `ticker + "_rating"`, `ticker + "_outlook"`, `ticker + "_outlook_changed"`, `ticker + "_52w"`, `ticker + "_flags"`, `ticker + "_shares"` — each gets its own `random.Random` instance to avoid bleeding between fields.
- **MARKET_CAP_USD hybrid pattern.** `_market_cap` reads year-stable `log_shares` from the `_shares` salt stream, then multiplies `BASE_PRICE × shares × (1 + thirty_day_change/100)`. The thirty-day-change daily noise is what makes MARKET_CAP slightly different day-to-day; the underlying shares count stays year-stable. **Don't compute MARKET_CAP from a daily salt** — the year-stable shares-outstanding base is the load-bearing piece for plausibility (a real ticker's shares outstanding doesn't change daily).
- **LIQUIDITY_TIER uses the year-stable shares-outstanding base WITHOUT the daily noise** — re-derived inside `_liquidity_tier` to avoid tier flips on small daily MARKET_CAP movements. Don't tie LIQUIDITY_TIER to the noisy daily MARKET_CAP_USD; tie it to the year-stable base.
- **`_SECTOR_RATING_BIAS` covers 7 sectors + a default fallback.** `_DEFAULT_RATING_BIAS` handles None / unknown SECTOR (Healthcare-like distribution clustered Baa1/Baa2). The L1 fixture's `UNKN` instrument exercises the default-fallback path.
- The `write_pandas(auto_create_table=True)` mis-types `datetime64[ns]` as `NUMBER(38,0)`. The MERGE source SELECT casts back via `TO_TIMESTAMP_NTZ(GENERATED_AT::NUMBER / 1000000000)` and `TO_TIMESTAMP_NTZ(LAST_DATA_REFRESH_AT::NUMBER / 1000000000)` — see template Task 4 §_merge.
- 13 columns total: 12 NOT NULL (TICKER, PROFILE_DATE, CREDIT_RATING, RATING_OUTLOOK, MARKET_CAP_USD, DAILY_VOLATILITY_PCT, THIRTY_DAY_PRICE_CHANGE_PCT, FIFTY_TWO_WEEK_HIGH_PRICE, FIFTY_TWO_WEEK_LOW_PRICE, RATING_AGENCY_FLAG_COUNT, LIQUIDITY_TIER, LAST_DATA_REFRESH_AT, GENERATED_AT) + 1 NULLable (OUTLOOK_LAST_CHANGED_DATE).
- L1 conftest defines a synthetic 10-instrument `SAMPLE_INSTRUMENTS` fixture covering 7 SECTOR values + 2 edge cases (TINY very-low BASE_PRICE; UNKN None SECTOR). **Cumulus_Common's `SAMPLE_ANCHORS` is unused** for Plan 13 — there are no account anchors involved.
- The DLO → DMO field mapping must be completed in DC Setup UI for fully-custom DMOs (the API endpoint returns 500). See the recipe at `../Snowflake_Claritas_Demographics/docs/dc-setup-recipe.md` from Plan 1 T7.
- Snowflake DATE columns auto-discover as `MM/dd/yyyy` in the DC data-stream POST body. Use that format for `PROFILE_DATE`'s and `OUTLOOK_LAST_CHANGED_DATE`'s `sourceFields` entries, NOT `yyyy-MM-dd`.
- DC mapping descriptions must be ≤521 chars (Plan 6 finding). Trim the DMO description if drafting against the rowspec verbatim.
- **No `ssot__AccountId__c`** in the DC field mapping — DMO is not joinable to `ssot__Account__dlm` (instrument-scoped, like Plan 4 was branch-scoped). Use `TICKER` (`ticker__c`, KQ qualifier) as the canonical join key for downstream queries; account-level joins go through the existing trades-pipeline `(account, ticker, position)` graph.
- DC PK collapses to `profileDate__c` + KQ on `ticker__c` (single-column-PK rule from Plan 4).
- Cadence is **daily** (`'USING CRON 0 1 * * * UTC'` — every day at 01:00 UTC, pre-Asia-open). Earlier than Plan 7's 06:00 UTC because Moody's market context is published pre-open in Tokyo (00:00 UTC = market open in Asia); 01:00 UTC gives a 30-min buffer after the (synthetic) data publication AND keeps the two daily plans clear of each other on the same warehouse.
- **Moody's apostrophe.** Verify the deploy script's identifier sanitizer handles `'`; strip in the build step if not. The directory and table names use `Moodys` / `MOODYS` (no apostrophe); only the human-facing description text carries the apostrophe.
- `accounts_processed = len(audience) = row count` (1:1 emit per instrument per day) — same shape as Plans 1-3, 5, 7, 8. **Note**: the column name is account-flavored but Plan 13 stores instrument count there.
- **Volume.** ~2,004 rows/day. Re-runs same day MERGE-replace in place; no daily history retained beyond `LAST_DATA_REFRESH_AT`. Live storage stays at ~2,004 rows year-round. Lower per-day volume than Plan 7 (36,813), but at daily cadence the cumulative-if-history-were-retained shape is ~730,000/year.
