# AGENTS.md — Snowflake_WorldCheck_AML

Synthetic LSEG World-Check / Dow Jones Risk & Compliance / ComplyAdvantage-style AML / sanctions / PEP screening dataset for the Cumulus FSC demo. One of 13. **Account-scoped with daily cadence — first daily-cadence dataset AND first all-accounts-audience dataset in the rollout.**

> **v1.x multi-org-additive (Phase A, 2026-05-29 commit `c9119d32`).** Table now leads with `ORG_ID VARCHAR(18) NOT NULL DEFAULT 'JDO'` as the first column; PK promoted from `(ACCOUNT_ID, PROFILE_DATE)` to `(ORG_ID, ACCOUNT_ID, PROFILE_DATE)` — note 1:1 emit shape (one row per anchor per screening day), not 1:N. SP row factory stamps `"ORG_ID": anchor.get("ORG_ID", "JDO")` as the first key; MERGE source SELECT, ON, INSERT lists all lead with ORG_ID; UPDATE SET deliberately skips ORG_ID (PK-component, immutable). Backward-compatible — JDO loaders continue working unchanged via DEFAULT. Multi-org rollout runbook: `Snowflake_Cumulus_Common/docs/ROLLOUT.md`.

## Boundaries
- Owns: `FINS.PUBLIC.WORLD_CHECK_AML`, `SP_GENERATE_WORLD_CHECK_AML`, `TASK_DAILY_WORLD_CHECK_AML`, and the DC Data Stream / DLO / DMO that federates this table.
- Does NOT own: `V_ACCOUNT_ANCHORS`, `MASTER_ACCOUNTS`, the seed/coverage helpers — see `Snowflake_Cumulus_Common`.
- Does NOT own any outbound Snowflake share. DC reads through via the existing "Snowflake (Federate / Zero Copy)" connector.
- **Account-scoped** — rows are keyed by composite PK `(ACCOUNT_ID, PROFILE_DATE)`, with FK `ACCOUNT_ID` to `ssot__Account__dlm`. **1:1 — one row per distinct account per screening day.**

## Conventions
- The SP uses `cumulus_common.seed_for(...)` for determinism with salt `"worldcheck"` (**day-bucketed**, not month — daily cadence).
- **Three independent seed streams per anchor per run:**
  - **Main daily seed** — `seed_for(account_id, "worldcheck", day_start)` where `day_start = run_ts.replace(hour=0,minute=0,second=0,microsecond=0)`. Drives component-flag draws (sanctions / PEP / adverse-media) + adverse-media categories + overall-rating noise tail. Mid-day re-runs are byte-identical because `day_start` truncates the time component.
  - **Year-stable jurisdiction seed** — `seed_for(account_id + "_jurisdiction", "worldcheck_jurisdiction", datetime(run_ts.year, 1, 1))`. Drives RISK_JURISDICTION_CODE + RISK_JURISDICTION_TIER. A customer's primary jurisdiction doesn't change day-to-day; this keeps it stable across the calendar year.
  - **Year-stable case seed** — `seed_for(account_id + "_case", "worldcheck_case", datetime(run_ts.year, 1, 1))`. Drives CASE_REFERENCE. Once a case is opened, the vendor case ID stays the same year-round.
- Plan 5 had **one** year-stable salt (`treasury_year`); Plan 7 has **two** (`worldcheck_jurisdiction` + `worldcheck_case`).
- The SP uses `cumulus_common.assert_coverage(session, expected_sql, actual_sql)` for the standard 1:1 audience-vs-actual coverage check (every distinct anchor must produce exactly one row per screening day).
- The MERGE replaces on composite PK `(ACCOUNT_ID, PROFILE_DATE)`. Re-runs same calendar day are idempotent (byte-identical). Re-runs on a later day INSERT new rows for that day's screening.
- Audience SQL is `SELECT DISTINCT * FROM V_ACCOUNT_ANCHORS` (**no WHERE clause** — first all-accounts plan). 36,813 distinct accounts (37,445 raw with 1.7% MASTER_ACCOUNTS dups; the `DISTINCT` collapses them).
- 30-jurisdiction pool with explicit Standard / Enhanced / Prohibited tiers — see rowspec `_PROHIBITED_JURISDICTIONS`, `_ENHANCED_JURISDICTIONS`, `_STANDARD_JURISDICTIONS`. NOT license-grade vendor data.
- `accounts_processed` in `TASK_EXECUTION_LOG` equals `len(audience)` — the **distinct customer count** AND the row count (1:1 emit per anchor per day).

## Tests
- L1 (pytest): determinism on the same calendar day (byte-identical at 03:00 vs 23:30), input-shape validation (ACCOUNT_ID non-empty), boring case (clean Retail anchor → Low rating, US jurisdiction, all flags false, NULL categories, NULL case ref, Unchanged delta), **anchor-independent bias** (the row factory reads ONLY `account_id`; verify two anchors with identical IDs but different BIRTHDATE/income/state produce identical rows — only ACCOUNT_ID matters), day-to-day stable jurisdiction (same RISK_JURISDICTION_CODE on May 28 and May 29), population-rate convergence (sanctions ~0.5%, PEP ~1.2%, adverse-media ~3.0% within ±0.3 pp across SAMPLE_ANCHORS × 365 days), CHANGE_SINCE_LAST_RUN coherence (~99% Unchanged on day-2 run), CASE_REFERENCE year-stable for High/Severe accounts, schema contract.
- L2 (`tests/integration/`): deploys SP into a fixture-backed schema with `V_ACCOUNT_ANCHORS_FIXTURE` (10 PERSON + 4 BUSINESS — **no filter, all in audience**); asserts coverage (distinct accts = 14 = total rows), idempotency on same-day re-run (ROWS_INSERTED=0), all 13 columns populated correctly, day-2 re-run (`run_ts` shifted +1 day) emits 14 new rows with non-Unchanged CHANGE_SINCE_LAST_RUN visible on at least one anchor.
- L3 (manual smoke, post-deploy): one CALL against jdo-uqj0jr; row count = 36,813 (distinct accounts); 5-row sample plausibility (mix of Low / Medium / High / Severe); rating distribution (~92% Low, 6% Medium, 1.7% High, 0.3% Severe within ±1 pp); component flag rates (sanctions ~0.5%, PEP ~1.2%, adverse-media ~3.0%); jurisdiction-tier distribution (~98.5% Standard, ~1.0% Enhanced, ~0.5% Prohibited); rollup invariants (all Severe rows have `RISK_JURISDICTION_TIER='Prohibited'` OR `SANCTIONS_HIT=true`; all High/Severe rows have `CASE_REFERENCE` populated; all non-flagged rows have `ADVERSE_MEDIA_CATEGORIES` NULL); day-2 re-run shape (~99% Unchanged, CASE_REFERENCE stable for accounts that are High/Severe both days).

## Gotchas
- **Daily cadence.** `PROFILE_DATE` is a DATE column equal to `day_start.date()`, NOT `PROFILE_MONTH`/`PROFILE_QUARTER`. The seed bucket is day-start `datetime(year, month, day, 0, 0, 0)` — mid-day re-runs are byte-identical because the time component is truncated. Plans 1-3, 6 use month-bucketing; Plan 5 uses quarter-bucketing; Plan 7 introduces day-bucketing for the first time.
- **All-accounts audience.** No `WHERE` predicate beyond `SELECT DISTINCT *`. **`_anchor_in_audience(anchor)` always returns `True`** — there is no filter cohort. Don't reintroduce a CLIENT_CATEGORY or ACCOUNT_TYPE_FLAG check; AML is non-negotiable for every customer (legal posture). Plan 4's `POSTAL_CODE <> ''` filter doesn't apply; Plan 5's PERSON-only filter doesn't apply; Plan 6's `CLIENT_CATEGORY IN (...)` filter doesn't apply.
- **Anchor-independent bias.** The row factory reads ONLY `anchor["ACCOUNT_ID"]` — NO BIRTHDATE / ANNUAL_INCOME / STATE / CLIENT_CATEGORY / POSTAL_CODE / ZIP. Even `RISK_JURISDICTION_CODE` is **synthesized** (year-stable salt `worldcheck_jurisdiction`) instead of read from the dirty `anchor.COUNTRY_CODE` field (96% `US` + 30% empty/dirty per Plan 0 probe — no signal). This is a structural deviation from Plans 1-6 where the row factory inspects multiple anchor fields for bias. Don't reintroduce demographic reads here — they are not available signals.
- **Two year-stable salts.** `worldcheck_jurisdiction` for RISK_JURISDICTION_CODE+TIER (a customer's primary jurisdiction doesn't shift day-to-day); `worldcheck_case` for CASE_REFERENCE (once a case is opened, the vendor case ID stays the same year-round). Both bucketed by `datetime(run_ts.year, 1, 1)`. Plan 5 has one year-stable salt; Plan 7 has two — keep them separate so the per-account jurisdiction RNG stream doesn't bleed into the case-ID RNG stream.
- **3 BOOLEAN columns.** `SANCTIONS_HIT`, `PEP_HIT`, `ADVERSE_MEDIA_HIT` — all NOT NULL. DC needs explicit `Boolean` declaration in the DLO + DMO field input for ALL THREE (Plan 5 finding — defaulting to Text fails the Boolean parse). Snowflake stores BOOLEAN natively; the MERGE source SELECT casts via `SANCTIONS_HIT::BOOLEAN`, `PEP_HIT::BOOLEAN`, `ADVERSE_MEDIA_HIT::BOOLEAN` to be explicit (write_pandas could mis-type on an empty target).
- **2 NULLable columns.** `ADVERSE_MEDIA_CATEGORIES` is NULL when `ADVERSE_MEDIA_HIT=false` (~97% of rows). `CASE_REFERENCE` is NULL when `OVERALL_RISK_RATING NOT IN ('High', 'Severe')` (~98% of rows). The DDL marks both as NULL; the MERGE source SELECT must preserve `None` → `NULL`.
- **CHANGE_SINCE_LAST_RUN re-derives yesterday's seed.** The day-to-day delta is computed by seeding a second `random.Random` with `seed_for(account_id, "worldcheck", run_ts - 1 day)` and re-running the component-flag draws + the rating rollup. This means the SP body must include `_sanctions_hit`, `_pep_hit`, `_adverse_media_hit`, `_overall_risk_rating`, and `_risk_jurisdiction_stable` as **pure functions** so they can be invoked twice (once for today, once for yesterday) with separate `random.Random` instances. Don't capture mutable state in closures here.
- Salt is `"worldcheck"` (daily cadence) for the main row; `"worldcheck_jurisdiction"` and `"worldcheck_case"` for year-stable fields.
- Cadence is **daily** (`'USING CRON 0 6 * * * UTC'` — 06:00 UTC daily, after LSEG's overnight feed publishes at ~02:00 GMT).
- The `write_pandas(auto_create_table=True)` mis-types `datetime64[ns]` as `NUMBER(38,0)`. The MERGE source SELECT casts back via `TO_TIMESTAMP_NTZ(GENERATED_AT::NUMBER / 1000000000)` and `TO_TIMESTAMP_NTZ(LAST_SCREENED_AT::NUMBER / 1000000000)` — see template Task 4 §_merge.
- 13 columns total: 11 NOT NULL (ACCOUNT_ID, PROFILE_DATE, OVERALL_RISK_RATING, SANCTIONS_HIT, PEP_HIT, ADVERSE_MEDIA_HIT, RISK_JURISDICTION_CODE, RISK_JURISDICTION_TIER, LAST_SCREENED_AT, CHANGE_SINCE_LAST_RUN, GENERATED_AT) + 2 NULLable (ADVERSE_MEDIA_CATEGORIES, CASE_REFERENCE).
- L1 conftest provides a synthetic anchor fixture — `SAMPLE_ANCHORS` from Cumulus_Common with `in_audience_anchors = all_anchors` (everyone is in audience; no filter step in the L1 helper).
- The DLO → DMO field mapping must be completed in DC Setup UI for fully-custom DMOs (the API endpoint returns 500). See the recipe at `../Snowflake_Claritas_Demographics/docs/dc-setup-recipe.md` from Plan 1 T7.
- Snowflake DATE columns auto-discover as `MM/dd/yyyy` in the DC data-stream POST body. Use that format for `PROFILE_DATE`'s `sourceFields` entry, NOT `yyyy-MM-dd`.
- DC mapping descriptions must be ≤521 chars (Plan 6 finding). Trim the DMO description if drafting against the rowspec verbatim.
- `accounts_processed = len(audience)` matches row count (1:1 emit per anchor per day) — unlike Plan 6 where `accounts_processed ≠ row_count` (1:N).
- **Volume.** ~36,813 rows/day. Re-runs same day MERGE-replace in place; no daily history retained beyond `LAST_SCREENED_AT` + `CHANGE_SINCE_LAST_RUN`. Live storage stays at ~36,813 rows year-round.
