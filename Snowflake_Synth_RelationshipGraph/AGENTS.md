# AGENTS.md — Snowflake_Synth_RelationshipGraph

Synthesized directed relationship-graph dataset for the Cumulus FSC demo. One of 13. **All-accounts audience with weekly cadence — the second 1:N dataset (Plan 6 was first), the second weekly cadence (Plan 6 first; Plan 12 third), and the only edge-scoped dataset in the rollout. First Cumulus dataset with cross-plan SOFT dependencies (Claritas / D&B / BoardEx).**

> **v1.x multi-org-additive (Phase A, 2026-05-29 commit `c9119d32`).** Table now leads with `ORG_ID VARCHAR(18) NOT NULL DEFAULT 'JDO'` as the first column; PK promoted from `(SRC_ACCOUNT_ID, DST_ACCOUNT_ID, EDGE_TYPE)` to `(ORG_ID, SRC_ACCOUNT_ID, DST_ACCOUNT_ID, EDGE_TYPE)`. **ORG_ID is stamped on every emitted edge dict** (HOUSEHOLD, CORPORATE_PARENT, BOARD_MEMBER, ADVISOR_BOOK, REFERRAL, BUSINESS_OWNER, SELF) — `"ORG_ID": anchor.get("ORG_ID", "JDO")` as the first key. **Cross-org edges are not modeled in Phase A** — every edge's SRC and DST share the same ORG_ID, scoped through the SRC anchor. MERGE source SELECT, ON, INSERT lists all lead with ORG_ID; UPDATE SET deliberately skips ORG_ID (PK-component, immutable). Backward-compatible — JDO loaders continue working unchanged via DEFAULT. Multi-org rollout runbook: `Snowflake_Cumulus_Common/docs/ROLLOUT.md`.

## Boundaries
- Owns: `FINS.PUBLIC.SYNTH_RELATIONSHIP_GRAPH`, `SP_GENERATE_SYNTH_RELATIONSHIP_GRAPH`, `TASK_WEEKLY_SYNTH_RELATIONSHIP_GRAPH`, and the DC Data Stream / DLO / DMO that federates this table.
- Does NOT own: `V_ACCOUNT_ANCHORS`, `MASTER_ACCOUNTS`, the seed/coverage helpers — see `Snowflake_Cumulus_Common`.
- Does NOT own any outbound Snowflake share. DC reads through via the existing "Snowflake (Federate / Zero Copy)" connector.
- Does NOT own the upstream cross-plan tables (`CLARITAS_DEMOGRAPHICS`, `DNB_BUSINESS_CREDIT`, `BOARDEX_EXEC_INTEL`) — those are Plans 1, 3, 10. Plan 9 reads them via SOFT-existence probes and silently skips edge types whose source tables are absent.
- **Edge-scoped** — rows are keyed by composite PK `(SRC_ACCOUNT_ID, DST_ACCOUNT_ID, EDGE_TYPE)`. SRC_ACCOUNT_ID is the anchor (FK to `ssot__Account__dlm`); DST_ACCOUNT_ID is the destination account (or self for SELF edges). **1–N edge rows per anchor per week.**

## Conventions
- The SP uses `cumulus_common.seed_for(...)` for determinism with salt `"synth-graph"` (**week-bucketed**, single-salt — Plan 6's simpler shape).
- **One seed stream per anchor per run** — `seed_for(account_id, "synth-graph", week_start)` where `week_start = run_ts.replace(day=run_ts.day - run_ts.weekday(), hour=0, minute=0, second=0, microsecond=0)`. Drives EDGE_WEIGHT, CONFIDENCE_PCT, EDGE_DISCOVERED_DATE / EDGE_LAST_SEEN_DATE, sparse-edge dice rolls (REFERRAL ~5%, BUSINESS_OWNER ~2-3%, ADVISOR_BOOK ~30% gate). Mid-week re-runs are byte-identical because `week_start` truncates day-to-Monday + time.
- The SP uses `cumulus_common.assert_coverage(session, expected_sql, actual_sql)` for the **distinct-src-accounts** half of coverage. Single-part assertion only — no row-count band, because the SELF-fallback guarantees `COUNT(DISTINCT SRC_ACCOUNT_ID) = audience_size` regardless of which cross-plan tables are populated.
- The MERGE replaces on composite PK `(SRC_ACCOUNT_ID, DST_ACCOUNT_ID, EDGE_TYPE)`. Re-runs same calendar week are idempotent (byte-identical). Re-runs in a later week INSERT new edge state for that week (and possibly delete now-absent edges, depending on MERGE shape).
- **Rows are sorted by `(DST_ACCOUNT_ID, EDGE_TYPE)` before MERGE** — deterministic ordering across re-runs (the multi-edge-function flat-extend means natural emission order is non-deterministic without a pre-MERGE sort). **Different sort key from Plan 6** (which sorted by `HELD_AWAY_ACCOUNT_ID`).
- Audience SQL is `SELECT DISTINCT * FROM V_ACCOUNT_ANCHORS` — `1=1` predicate, all accounts. 36,813 distinct anchors (probed 2026-05-28; 37,445 raw rows due to 1.7% MASTER_ACCOUNTS dup, defensively dedup'd until upstream fix). Anchor reads: `ACCOUNT_ID`, `ACCOUNT_TYPE_FLAG`, `CLIENT_CATEGORY`, `ANNUAL_INCOME`, `BIRTHDATE`. None NULL on the live audience for the in-use fields.
- `accounts_processed` in `TASK_EXECUTION_LOG` is `len(audience)` — the **distinct customer count** (~36,813), NOT `len(records)` (~110K-180K). The row count goes in a separate audit field (or into the SP's RuntimeError messages and the per-edge-type emit-counts string written to `TASK_EXECUTION_LOG.ERROR_MESSAGE` on success).
- **Per-edge-type emit counts logged to `TASK_EXECUTION_LOG.ERROR_MESSAGE` on success** (e.g. `"emit_counts: HOUSEHOLD=24013, CORPORATE_PARENT=2891, BOARD_MEMBER=skipped(no_table), ADVISOR_BOOK=2347, REFERRAL=1764, BUSINESS_OWNER=687, SELF=4899"`). Operator-visible signal for which SOFT dependencies were satisfied this run.

## Tests
- L1 (pytest): determinism on multi-row output (re-running `_rows_for` with the same `available_edge_types` set returns identical sorted list), input-shape validation (ACCOUNT_ID non-empty), boring case (every anchor whose `available_edge_types == {"SELF"}` returns exactly 1 row with EDGE_TYPE='SELF', SRC=DST, EDGE_WEIGHT=1.000, CONFIDENCE_PCT=100.00), **per-anchor SELF-fallback invariant** (no anchor returns `[]` — ever), **per-row edge-weight bounds** (every emitted row's EDGE_WEIGHT is in the `_EDGE_TYPE_WEIGHT_BAND[edge_type]` band; SELF exactly 1.000; CORPORATE_PARENT and BUSINESS_OWNER ≥ 0.85), **per-row confidence bounds** (CONFIDENCE_PCT in [30.00, 100.00]; SELF exactly 100.00), **per-row date-coherence invariants** (EDGE_LAST_SEEN_DATE ≥ EDGE_DISCOVERED_DATE; EDGE_DISCOVERED_DATE ≤ run_ts.date(); SELF rows: discovered = last_seen = week_start.date()), **edge-count band per-anchor** (mean edges/anchor across SAMPLE_ANCHORS in [1, 5] — inclusive band; with L1's empty cross-plan tables the mean will be 1.0, with full live data ~3-5), schema contract.
- L2 (`tests/integration/`): deploys SP into a fixture-backed schema with `V_ACCOUNT_ANCHORS_FIXTURE` (14 mixed Retail/Wealth/Business); asserts coverage (`COUNT(DISTINCT SRC_ACCOUNT_ID) = 14`), idempotency on same-week re-run (ROWS_INSERTED=0), at least one anchor produces ≥1 row (every anchor does, by SELF-fallback), `COUNT(*) WHERE EDGE_TYPE='SELF' = 14` (with empty cross-plan fixtures, every anchor SELF-falls-back). **SOFT-dependency path test**: install a mock `CLARITAS_DEMOGRAPHICS` table with 2 fixture rows, re-run, verify HOUSEHOLD edges appear and SELF count drops accordingly.
- L3 (manual smoke, post-deploy): one CALL against jdo-uqj0jr; `COUNT(DISTINCT SRC_ACCOUNT_ID) = 36,813` (audience size; SELF-fallback guarantees this regardless of cross-plan availability); total rows in band [36,813, 200,000]; per-EDGE_TYPE distribution matches the per-edge-type emit-counts logged in TASK_EXECUTION_LOG.ERROR_MESSAGE; SELF-edge invariant `COUNT(*) WHERE EDGE_TYPE='SELF' AND (SRC != DST OR EDGE_WEIGHT != 1.000 OR CONFIDENCE_PCT != 100.00) = 0`; cross-plan visibility (BOARD_MEMBER count is either 0 if Plan 10 unshipped, or matches BoardEx director-seat count); 5-row sample plausibility per edge type.

## Gotchas
- **First edge-scoped dataset.** Row identity is the directed-edge tuple `(SRC, DST, EDGE_TYPE)` — NOT an account-anchored single key with a slot index. SRC_ACCOUNT_ID is the anchor (FK side); DST_ACCOUNT_ID is any other account in the audience (or the same account for SELF). HOUSEHOLD is the only bidirectional edge type — both `(A→B, HOUSEHOLD)` and `(B→A, HOUSEHOLD)` rows are emitted. All other non-SELF types are unidirectional.
- **Cross-plan SOFT dependencies — never fail the SP on a missing upstream table.** `main()` runs `_probe_available_edge_types(session)` at top-of-run. Each `IF EXISTS` probe (`SELECT 1 FROM <table> LIMIT 1`) is wrapped in try/except; absent tables silently filter out the corresponding edge_type from `available_edge_types`. `SELF`, `ADVISOR_BOOK`, `REFERRAL`, `BUSINESS_OWNER` are always-on; `HOUSEHOLD` requires `CLARITAS_DEMOGRAPHICS`; `CORPORATE_PARENT` requires `DNB_BUSINESS_CREDIT`; `BOARD_MEMBER` requires `BOARDEX_EXEC_INTEL`. Per manifest §3.2: "If any are absent, the corresponding edge type is skipped (not failed)." Plan 10 (BoardEx) is expected to ship after Plan 9, so BOARD_MEMBER edges may be absent at first deploy — this is the design, not a bug.
- **SELF self-edge fallback — the load-bearing coverage guarantee.** Every `_rows_for(anchor, ...)` that would otherwise return `[]` instead returns `[self_edge(anchor)]`. The SELF row has fixed `EDGE_WEIGHT=1.000`, `CONFIDENCE_PCT=100.00`, `EDGE_DISCOVERED_DATE=EDGE_LAST_SEEN_DATE=week_start.date()`, `METADATA=NULL`. This guarantees `COUNT(DISTINCT SRC_ACCOUNT_ID) = audience_size` always holds, even when every cross-plan edge source is empty or absent. Coverage assertion is therefore single-part (no row-count band).
- **Single salt, week-bucketed seed.** Salt is `"synth-graph"` (matches manifest). `seed_for(account_id, "synth-graph", week_start)` where `week_start` floors `run_ts` to Monday 00:00 UTC. Re-running the SP mid-week (e.g. Wednesday) produces byte-identical output to the Monday run because the seed and `GENERATED_AT` both anchor on the week's Monday. Plan 5 had 1 year-stable salt; Plan 7 had 3; Plan 8 reverted to single-salt; Plan 9 also single-salt — there is no year-stable subfield.
- **Per-anchor invariants, not distributional.** L1 unit-test environment lacks the cross-plan tables (Claritas/D&B/BoardEx), so `available_edge_types` collapses to `{ADVISOR_BOOK, REFERRAL, BUSINESS_OWNER, SELF}` plus most anchors land on SELF-fallback. Property tests target SELF invariants, weight/confidence bounds per emitted row, and date-coherence — NOT edge-type rate distributions. Distributional rate testing is meaningless when most edge types resolve to "soft-skip". Plans 1-7 used rate-convergence tests over SAMPLE_ANCHORS — Plans 8 and 9 are the first datasets where the structural shift to per-anchor invariants is load-bearing.
- **Composite 3-column natural PK collapses to single-column DC PK.** Snowflake DDL keeps `PRIMARY KEY (SRC_ACCOUNT_ID, DST_ACCOUNT_ID, EDGE_TYPE)`. DC enforces single-column DMO PK (Plan 4 finding); we project a derived `EDGE_ID = SUBSTR(SHA2(SRC_ACCOUNT_ID || '|' || DST_ACCOUNT_ID || '|' || EDGE_TYPE, 256), 1, 32)` at the DLO source view level (`V_SYNTH_RELATIONSHIP_GRAPH_FOR_DC`) and use `edgeId__c` as the single-column DMO PK with KQ qualifiers `srcAccountId__c` (FK to `ssot__Account__dlm`), `dstAccountId__c`, `edgeType__c`. **Do not add an EDGE_ID column to the Snowflake table itself** — it lives only in the DLO source view.
- **EDGE_TYPE is `VARCHAR(20)`** in Snowflake — fits the seven values (HOUSEHOLD / CORPORATE_PARENT / BOARD_MEMBER / ADVISOR_BOOK / REFERRAL / BUSINESS_OWNER / SELF). `CORPORATE_PARENT` is the longest at 16 chars.
- **EDGE_WEIGHT is `NUMBER(4,3)`** — range 0.000–1.000, three-decimal precision per the rowspec band table. SELF is exactly 1.000.
- **CONFIDENCE_PCT is `NUMBER(5,2)`** — range 0.00–100.00 (clamped in code to [30.00, 99.99] for non-SELF, exactly 100.00 for SELF). Two-decimal precision.
- **METADATA is `VARCHAR(500) NULL`** — JSON-ish key-value notes, only populated when synthesis adds context (e.g. `'{"household_role":"member"}'`, `'{"board_role":"Director"}'`, `'{"link_source":"dnb"}'`, `'{"ownership_pct":"majority"}'`). NULL for SELF edges and for edge types that don't add context. This is the only NULLable column.
- **Status-first computation in `_make_edge`.** EDGE_DISCOVERED_DATE is computed first; CONFIDENCE_PCT reads it for age-decay (`age_days = (run_date - edge_discovered_date).days`); EDGE_LAST_SEEN_DATE is then derived to satisfy `last_seen >= discovered`. Don't reorder — the age-decay calculation depends on the discovered-date being already set.
- **Anchor reads.** `ACCOUNT_ID`, `ACCOUNT_TYPE_FLAG`, `CLIENT_CATEGORY`, `ANNUAL_INCOME`, `BIRTHDATE` — none NULL on the live audience for the in-use fields. `BUSINESS_OWNER` filters on `ACCOUNT_TYPE_FLAG == 'PERSON'` AND `ANNUAL_INCOME >= 200_000`. `CORPORATE_PARENT` filters on `ACCOUNT_TYPE_FLAG == 'BUSINESS'`. `ADVISOR_BOOK` filters on `CLIENT_CATEGORY == 'Wealth Management'`.
- **Date-coherence invariants** (must hold per row, asserted in L1 and L3):
  - `EDGE_LAST_SEEN_DATE >= EDGE_DISCOVERED_DATE` — chronological coherence.
  - `EDGE_DISCOVERED_DATE <= run_ts.date()` — no future-dated discovery.
  - SELF rows: `EDGE_DISCOVERED_DATE == EDGE_LAST_SEEN_DATE == _week_start(run_ts).date()`.
- The `write_pandas(auto_create_table=True)` mis-types `datetime64[ns]` as `NUMBER(38,0)`. The MERGE source SELECT casts back via explicit `EDGE_TYPE::VARCHAR(20)`, `EDGE_WEIGHT::NUMBER(4,3)`, `CONFIDENCE_PCT::NUMBER(5,2)`, `METADATA::VARCHAR(500)`, `GENERATED_AT::TIMESTAMP_NTZ(9)` and `TO_TIMESTAMP_NTZ(GENERATED_AT::NUMBER / 1000000000)` if write_pandas surprises — see template Task 4 §_merge.
- 9 columns total: 8 NOT NULL + 1 NULLable (METADATA).
- L1 conftest provides a synthetic anchor fixture — `SAMPLE_ANCHORS` from Cumulus_Common with `in_audience_anchors = list(all_anchors)` (since audience is `1=1`, all sample anchors are in audience). `available_edge_types` defaults to `{"SELF"}` in unit tests (no cross-plan tables present); explicit per-test override available for testing the SOFT-dependency path.
- The DLO → DMO field mapping must be completed in DC Setup UI for fully-custom DMOs (the API endpoint returns 500). See the recipe at `../Snowflake_Claritas_Demographics/docs/dc-setup-recipe.md` from Plan 1 T7.
- Snowflake DATE columns auto-discover as `MM/dd/yyyy` in the DC data-stream POST body. Use that format for `EDGE_DISCOVERED_DATE`, `EDGE_LAST_SEEN_DATE`'s `sourceFields` entries, NOT `yyyy-MM-dd`.
- DC mapping descriptions must be ≤521 chars (Plan 6 finding). Trim the DMO description if drafting against the rowspec verbatim.
- `accounts_processed = len(audience) ≠ row count` (1:N emit per anchor per week). This is the third dataset where these diverge (Plan 6 was first, Plan 9 second 1:N — but Plan 9 is the first with cross-plan-gated edge counts).
- Cadence is **weekly** (`'USING CRON 0 5 * * 1 UTC'` — Monday at 05:00 UTC). **Second weekly cadence** in the rollout (Plan 6 was first; Plan 12 will be third). Same Monday 05:00 UTC slot as Plan 6 — tasks run sequentially in alphabetical-ish task-name order (TASK_WEEKLY_PLAID_HELD_AWAY before TASK_WEEKLY_SYNTH_RELATIONSHIP_GRAPH), not concurrently, so no warehouse contention concern.
- **Volume.** ~110K–180K rows/week — largest Cumulus table by 2-3× over Plan 6 (52K). With Plan 10 (BoardEx) absent at deploy time, expect ~80K-120K. Floor is `audience_size = 36,813` rows (every anchor SELF-falls-back, worst case). write_pandas chunking (default 16K) is fine; Plan 6's pattern carries over unchanged.
