# Integration guide — DC_AgentForce_Markdown_Renderer

How to wire the markdown renderer into different Salesforce surfaces. This component is built primarily for **Agentforce GenAI Function output**, but the underlying mechanism (Lightning Type with a renderer override) can be reused in several other contexts.

---

## TL;DR

- **The component renders any value typed `c__markdownResponse`** as styled HTML.
- **The wiring is declarative**, not procedural — there's no Flow plumbing or Apex sanitizer to invoke. Type the field, the platform binds the renderer.
- **Deploy this project FIRST**, then deploy any consumer that references `c__markdownResponse`.

---

## Surface 1: Agentforce GenAI Function (primary use case)

This is the canonical wiring. Any `GenAiFunction` whose output schema types `promptResponse` as `c__markdownResponse` will be auto-routed through `c/markdownRenderer` in the Agentforce desktop panel.

### Step 1 — Author the GenAiFunction

Folder layout in your project:

```
force-app/main/default/genAiFunctions/My_Markdown_Function/
├── My_Markdown_Function.genAiFunction-meta.xml
├── input/
│   └── schema.json
└── output/
    └── schema.json
```

### Step 2 — Type the output as `c__markdownResponse`

`output/schema.json`:

```json
{
  "unevaluatedProperties": false,
  "properties": {
    "promptResponse": {
      "title": "Prompt Response",
      "description": "Markdown-formatted response.",
      "lightning:type": "c__markdownResponse",
      "lightning:isPII": false,
      "copilotAction:isDisplayable": true,
      "copilotAction:isUsedByPlanner": true
    }
  },
  "lightning:type": "lightning__objectType"
}
```

The `c__` prefix is the namespace — leave it as `c__` if you have no managed-package namespace. **Do not change `markdownResponse`** — that's the developer name registered by this project's `lightningTypes/markdownResponse/schema.json`.

### Step 3 — Reference from the Agent Script

If the GenAiFunction is dispatched from an Agent Script subagent, the output binding must echo the same Lightning type:

```yaml
actions:
    My_Markdown_Function:
        description: |
            What this function does.
        label: "My Markdown Function"
        require_user_confirmation: False
        include_in_progress_indicator: True
        source: "My_Markdown_Function"
        target: "generatePromptResponse://My_Retriever_or_Prompt"
        inputs:
            "Input:Question": string
                description: |
                    User question text.
                label: "Question"
                is_required: True
                is_user_input: True
        outputs:
            "promptResponse": object
                complex_data_type_name: "c__markdownResponse"   # <- THIS LINE
                description: |
                    Prompt response generated by the action.
                label: "Prompt Response"
                is_displayable: True
                filter_from_agent: False
```

The `complex_data_type_name: "c__markdownResponse"` line is what triggers the Agentforce panel to look up the type's renderer and route through `c/markdownRenderer`. Without it, the response renders as raw text.

### Step 4 — Deploy in this order

```bash
# 1. Renderer (creates the Lightning type)
cd DC_AgentForce_Markdown_Renderer
sf project deploy start --source-dir force-app --target-org my-org --wait 10

# 2. Your consumer (references the type)
cd ../My_Agent_Project
sf project deploy start --source-dir force-app --target-org my-org --wait 10
```

If the consumer deploys first, the type lookup fails with `c__markdownResponse not found`. Recoverable — just deploy the renderer and re-run.

`Cumulus_Assistant/` is the live consumer. See:

- `Cumulus_Assistant/force-app/main/default/genAiFunctions/DC_Product_Offers/output/schema.json` — types `promptResponse`
- `Cumulus_Assistant/force-app/main/default/aiAuthoringBundles/Cumulus_Assistant/Cumulus_Assistant.agent` line ~3303 — `complex_data_type_name: "c__markdownResponse"`

## Surface 2: Custom Lightning component on a record/app/home page

The LWC's `js-meta.xml` currently only exposes `lightning__AgentforceOutput`. To use it directly on a record page, app page, or home page, **add the targets**:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<LightningComponentBundle xmlns="http://soap.sforce.com/2006/04/metadata">
    <apiVersion>66.0</apiVersion>
    <isExposed>true</isExposed>
    <masterLabel>Markdown Renderer</masterLabel>
    <targets>
        <target>lightning__AgentforceOutput</target>
        <target>lightning__RecordPage</target>
        <target>lightning__AppPage</target>
        <target>lightning__HomePage</target>
    </targets>
    <targetConfigs>
        <targetConfig targets="lightning__RecordPage,lightning__AppPage,lightning__HomePage">
            <property name="value"
                      label="Markdown text"
                      type="String"
                      description="Markdown-formatted text to render."/>
        </targetConfig>
    </targetConfigs>
</LightningComponentBundle>
```

Then drop it into App Builder and bind `value` to:
- A literal markdown string (for static content)
- A formula field that produces markdown
- A custom Lightning Type field on the record

The component accepts:
- A string (rendered as markdown)
- An object with `text` or `promptResponse` properties (Agentforce-shape)
- Any JSON-serializable value (falls back to `JSON.stringify`)

## Surface 3: Programmatic use from another LWC

Import and embed:

```html
<template>
    <c-markdown-renderer value={myMarkdown}></c-markdown-renderer>
</template>
```

```js
import { LightningElement } from 'lwc';

export default class MyParent extends LightningElement {
    myMarkdown = '## Hello\n\nThis is **bold**, *italic*, and [a link](https://example.com).';
}
```

Reactive: changing `myMarkdown` triggers `renderedCallback` and re-renders. Idempotent re-renders are short-circuited by the `innerHTML !== newHtml` guard.

## Surface 4: Other Lightning Types (advanced)

The pattern is reusable beyond Agentforce. To wire the renderer into a different platform surface that supports Lightning Type renderer overrides:

1. **Find the surface name.** Salesforce uses different surface names for different host UIs:
   - `lightningDesktopGenAi` — Agentforce desktop panel (used by this project)
   - `lightningDesktop` — record/app/home pages with typed Lightning fields
   - Others may exist for Mobile, Experience Cloud, etc. — check the platform release notes.

2. **Add a renderer override** under `lightningTypes/markdownResponse/<surfaceName>/renderer.json`:

```json
{
    "renderer": {
        "componentOverrides": {
            "$": {
                "definition": "c/markdownRenderer"
            }
        }
    }
}
```

The `$` selector means "the root of the value." For nested fields, use a JSON Pointer like `/properties/text`.

3. **Bind a field to type `c__markdownResponse`** on whatever schema lives in the surface (custom field, Object Manager, Data Cloud DMO field, etc.).

This project ships `lightningDesktopGenAi/renderer.json` only. Add more if you have the use case.

## Markdown syntax reference

What `parseMarkdown` understands today:

| Syntax | Renders as | Notes |
|--------|------------|-------|
| `# H1` `## H2` `### H3` | `<h1>` `<h2>` `<h3>` | Line-anchored, must be at start of line. |
| `**bold**` | `<strong>` | Lazy match; non-greedy. |
| `*italic*` | `<em>` | Lazy match; non-greedy. |
| `***both***` | `<strong><em>both</em></strong>` | Triple-asterisk handled before single/double. |
| `` `inline code` `` | `<code>` | Backticks; content escaped (no markdown inside). |
| ```` ```lang\ncode\n``` ```` | `<pre><code>` | Optional language tag (currently ignored). Content preserved verbatim. |
| `> quoted text` | `<blockquote>` | Multiple consecutive `> ` lines collapse into one block. |
| `- item` | `<ul><li>` | Hyphen + space. |
| `1. item` | `<ol><li>` | Digits + period + whitespace. |
| `[label](url)` | `<a>` | URL scheme allowlisted (see security). |
| `\| h1 \| h2 \|`<br>`\|---\|---\|`<br>`\| a \| b \|` | `<table>` | GitHub-style; requires header + separator + body. |
| Blank line | Paragraph break | Double newline in input. |

Not supported:
- Strikethrough (`~~text~~`) — easy to add: one regex
- Setext-style headings (`===` or `---` underlines) — uncommon in LLM output
- Reference-style links (`[label][1]` + `[1]: url`) — more parser work
- Footnotes, definition lists, task lists, emoji shortcodes
- HTML passthrough — intentionally blocked by `escapeHtml`

If your prompt template will emit anything in the "not supported" list, either change the prompt or extend `parseMarkdown` (and add Jest coverage).

## Security model (what the renderer guarantees)

The component treats the input as **semi-trusted**: it's LLM-generated, possibly influenced by retrieval results that could include adversarial content.

| Attack | Mitigation | Where in code |
|--------|------------|---------------|
| `<script>` injection | `escapeHtml` runs on all text BEFORE markdown regex emits any tag. | `markdownRenderer.js:131-134` |
| `javascript:` link URI | URL scheme allowlist: `http(s):`, `mailto:`, `/`, `#`, `./`. Anything else → `href="#"`. | `markdownRenderer.js:67-73` |
| `java\tscript:` padding | Strip whitespace + `0x00-0x1F` + DEL `0x7F` BEFORE scheme matching. | `markdownRenderer.js:69` |
| Reverse tabnabbing | `rel="noopener noreferrer"` on every emitted `<a>`. | `markdownRenderer.js:71` |
| Referer header leakage | `noreferrer` (above) prevents `Referer` exposure to externally-linked sites. | `markdownRenderer.js:71` |
| HTML inside code blocks | Code-block content is round-tripped through placeholders, then `escapeHtml`'d at restore time. Markdown inside code is preserved as literal text. | `markdownRenderer.js:25-37, 102-112` |

What it **does not** mitigate:
- **Phishing via plausible link text.** A markdown link `[Click here](https://evil.example)` renders as a clickable link with whatever label the LLM chose. The URL allowlist confirms scheme but not domain. If your trust model requires domain allowlisting, extend the link parser.
- **Markdown that looks like code but isn't fenced.** If the LLM emits raw HTML without a fence, `escapeHtml` neutralizes it — but it'll show as literal `&lt;tag&gt;` text rather than executing. That's the correct behavior; just be aware the rendering won't look like the LLM "intended".
- **Resource exhaustion via huge input.** No length cap. If your prompt template can return megabytes, add a cap before passing to the renderer.

## Common deployment scenarios

### Scenario A: I want to add this to a new agent

1. Deploy `DC_AgentForce_Markdown_Renderer/` to the org (one-time).
2. In your agent project, type any GenAiFunction's output `promptResponse` as `c__markdownResponse`.
3. In the `.agent` script, set `complex_data_type_name: "c__markdownResponse"` on the matching action's `outputs:` block.
4. Deploy. Done.

### Scenario B: I want to change how the rendered HTML looks

Edit `force-app/main/default/lwc/markdownRenderer/markdownRenderer.css`. The CSS is shadow-scoped — your changes only affect this component.

To match SLDS dark mode or a specific design system, switch the hard-coded colors (`#181818`, `#0070d2`, `#f3f3f3`) to SLDS design tokens (`var(--lwc-colorTextDefault, #181818)` etc.).

### Scenario C: I want to add a new markdown feature

1. Add the regex in `parseMarkdown` at the right ordering position. Order matters — see the numbered comments in the source.
2. Add CSS rules in `markdownRenderer.css` if the new tag needs styling.
3. Add Jest tests in `__tests__/markdownRenderer.test.js` covering the construct + edge cases (empty input, nested constructs, malformed input).
4. If the feature accepts any user-controlled URL or attribute, extend the security model accordingly. Document the new attack class + mitigation in this guide.

### Scenario D: I want to use this on a non-Agentforce surface

See "Surface 2" and "Surface 3" above. The fastest path is "Surface 3" (programmatic embedding from another LWC) — no metadata changes needed.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| Response renders as raw `**bold**` text | Output schema not typed `c__markdownResponse`, OR renderer project not deployed | Verify `output/schema.json` has `"lightning:type": "c__markdownResponse"`; deploy `DC_AgentForce_Markdown_Renderer/` to the org. |
| Deploy fails with "type c__markdownResponse not found" | Consumer deployed before renderer | Deploy `DC_AgentForce_Markdown_Renderer/` first, then re-run consumer deploy. |
| Component shows nothing | `value` is null/undefined, OR `markdownText` getter returned empty | Check the data binding; remember the getter unwraps `value.text`, `value.promptResponse`, or stringifies. |
| Links emit `href="#"` even for valid URLs | URL has unusual chars (control bytes, unicode whitespace) being stripped | Inspect the cleaned URL in the regex on `markdownRenderer.js:69`; the strip is intentional but may be too aggressive for niche URLs. |
| Markdown table renders as raw pipes | Table grammar doesn't match (missing separator row, header/body cell-count mismatch) | Verify the markdown matches GitHub-flavored table grammar exactly. |
| Component works on desktop but not on mobile | `lookbehind` regex no longer used (removed in current parser), but check for other ES2018 features | Modern parser uses no lookbehind; if you re-introduce it, target iOS 16.4+. |
