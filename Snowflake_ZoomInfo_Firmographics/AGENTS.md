# AGENTS.md — Snowflake_ZoomInfo_Firmographics

Synthetic ZoomInfo / DiscoverOrg / Crunchbase-style B2B company-level firmographics dataset for the Cumulus FSC demo. One of 13. **BUSINESS-scoped (same audience as Plans 2 MSCI and 3 D&B), monthly cadence — the most "boring" Plan structurally in the rollout.** Same audience predicate, cadence, and 1:1 row shape as Plans 2 and 3; the only structural deviations are defensive HQ-string projection and two distributionally-gated NULLable columns, both inherited from prior Plans.

## Boundaries
- Owns: `FINS.PUBLIC.ZOOMINFO_FIRMOGRAPHICS`, `SP_GENERATE_ZOOMINFO_FIRMOGRAPHICS`, `TASK_MONTHLY_ZOOMINFO_FIRMOGRAPHICS`, and the DC Data Stream / DLO / DMO that federates this table.
- Does NOT own: `V_ACCOUNT_ANCHORS`, `MASTER_ACCOUNTS`, the seed/coverage helpers — see `Snowflake_Cumulus_Common`.
- Does NOT own any outbound Snowflake share. DC reads through via the existing "Snowflake (Federate / Zero Copy)" connector.
- **Account-scoped** — rows are keyed by composite PK `(ACCOUNT_ID, PROFILE_MONTH)`, with FK `ACCOUNT_ID` to `ssot__Account__dlm`. **1:1 — one row per distinct BUSINESS account per month.**

## Conventions
- The SP uses `cumulus_common.seed_for(...)` for determinism with salt `"zoominfo"` (**month-bucketed**, single-salt — no year-stable streams; same shape as Plans 2/3/6/8, simpler than Plan 7's 3-salt arrangement).
- **One seed stream per anchor per run** — `seed_for(account_id, "zoominfo", month_start)` where `month_start = run_ts.replace(day=1, hour=0, minute=0, second=0, microsecond=0)`. Drives FOUNDED_YEAR, LINKEDIN_FOLLOWERS, TECH_STACK_FLAGS pool draws, LAST_DATA_REFRESH_DATE jitter, and the synth-fallback ZIP when raw is empty. Mid-month re-runs are byte-identical because `month_start` truncates day + time.
- The SP uses `cumulus_common.assert_coverage(session, expected_sql, actual_sql)` for the standard 1:1 audience-vs-actual coverage check (every distinct BUSINESS anchor must produce exactly one row per month).
- The MERGE replaces on composite PK `(ACCOUNT_ID, PROFILE_MONTH)`. Re-runs same calendar month are idempotent (byte-identical). Re-runs in a later month INSERT new rows for that month.
- Audience SQL is `SELECT DISTINCT * FROM V_ACCOUNT_ANCHORS WHERE ACCOUNT_TYPE_FLAG = 'BUSINESS'`. 12,021 distinct accounts (probed 2026-05-28). INDUSTRY, ANNUAL_REVENUE, EMPLOYEE_COUNT all populated for this cohort.
- `accounts_processed` in `TASK_EXECUTION_LOG` equals `len(audience)` — the **distinct customer count** AND the row count (1:1 emit per anchor per month).
- Coverage rule: **rows = audience** (1:1 monthly per BUSINESS account).

## Tests
- L1 (pytest): determinism in the same calendar month (byte-identical at 03:00 vs 23:30, day-1 vs day-15), input-shape validation (ACCOUNT_ID non-empty), boring case (Mid-market Manufacturing, 120 employees, $25M revenue → `EMPLOYEE_BAND='51-200'`, `REVENUE_BAND='$10M-$50M'`, NAICS `336411` / SIC `3711`, `WEBSITE_DOMAIN` derived from name), **per-anchor band-consistency invariants** (every anchor with `EMPLOYEE_COUNT ∈ [51,200]` → `EMPLOYEE_BAND == '51-200'` etc. across all 7 buckets; same for REVENUE_BAND across 6 buckets), **range invariants** (FOUNDED_YEAR ∈ [1900, run_ts.year], LINKEDIN_FOLLOWERS ∈ [0, 5_000_000], LAST_DATA_REFRESH_DATE within 90d window), **vocabulary invariants** (band values exact, NAICS regex `^\d{6}$`, SIC regex `^\d{4}$`), **defensive-string invariants** (every row: `len(HQ_COUNTRY_CODE)==2`, `len(HQ_STATE_CODE)==2`, `len(HQ_POSTAL_CODE)==5` and non-empty), **distributional NULL-rate checks** rolled over 3+ months for ~30 rows per cohort: `WEBSITE_DOMAIN` ~99.5% non-NULL and `TECH_STACK_FLAGS` ~90% non-NULL, schema contract (14 keys exact).
- L2 (`tests/integration/`): deploys SP into a fixture-backed schema with 14 fixture anchors (12 BUSINESS spanning the 7 EMPLOYEE_BAND × 6 REVENUE_BAND grid + 2 PERSON to verify audience filter excludes them); asserts `COUNT(DISTINCT ACCOUNT_ID) = 12`, `COUNT(DISTINCT EMPLOYEE_BAND) ≥ 5`, every row has `len(HQ_COUNTRY_CODE) = 2 AND len(HQ_STATE_CODE) = 2 AND len(HQ_POSTAL_CODE) = 5`, idempotent re-run yields ROWS_INSERTED=0.
- L3 (manual smoke, post-deploy): one CALL against jdo-uqj0jr; row count ~12,021 (matches Plans 2/3 BUSINESS cardinality plus ~600 row delta); BUSINESS over-count warning logged at step 1.5; 5-row sample plausibility; EMPLOYEE_BAND distribution skewed toward `1-10` (BUSINESS-misclassified Person-Account cohort default); defensive-string invariants hold (no row has `len(HQ_COUNTRY_CODE) != 2`, no row has empty HQ_POSTAL_CODE); WEBSITE_DOMAIN NULL-rate ~0.5%; TECH_STACK_FLAGS NULL-rate ~10%.

## Gotchas
- **Most "boring" Plan structurally in the Cumulus rollout.** Same audience predicate as Plans 2 and 3, same monthly cadence, same 1:1 row shape, same MERGE pattern. Only TWO structural deviations from the canonical template, both inherited from prior plans: defensive HQ-string projection (Plan 4 v1.5) and two distributionally-gated NULLables (lighter-weight than Plan 8's per-anchor enum-driven NULL semantics). Implementation should be the most mechanical Plan since 2/3.
- **BUSINESS over-count expected** (~12,021 instead of CRM ~5K). The SP warns (`accounts_processed > 10000`) but does NOT fail — the long-term fix is upstream `PersonBirthdate__c` backfill, not a view change. Same caveat as Plans 2 and 3. See spec §3 v1.2 finding #3.
- **Defensive string handling (per Plan 4 v1.5 findings) — load-bearing.** Three V_ACCOUNT_ANCHORS string columns require defensive projection in the SP — none can be passed through raw. Document why each rule exists so future maintainers don't "simplify" the projection back to a raw passthrough:
  - `COUNTRY_CODE` — the SP projects the literal `'US'` regardless of source value. Reason: 4 rows in V_ACCOUNT_ANCHORS carry `'USA'` / `'United States'` literals (v1.5 finding #5). The `HQ_COUNTRY_CODE VARCHAR(2)` DDL would otherwise fail to insert dirty-literal rows. Demo is US-only.
  - `POSTAL_CODE` — synth-fallback to a deterministic 5-digit ZIP from the seed bytes (`f"{int.from_bytes(seed[:3], 'big') % 100000:05d}"`) when raw is empty. Reason: 10,798 V_ACCOUNT_ANCHORS rows carry empty-string POSTAL_CODE (v1.5 finding #4). Guarantees every row carries a non-empty `HQ_POSTAL_CODE`.
  - `STATE_CODE` — fallback via a SP-local `_state_from_zip` helper (10-entry ZIP-first-digit → 2-char state mapping) when raw is blank or `len < 2`. Reason: same empty-string drift as POSTAL_CODE. Guarantees `len(HQ_STATE_CODE) == 2`.
  - The L1 tests assert these as **defensive string invariants** (property #4 in the L1 invariant set): `len(HQ_COUNTRY_CODE) == 2`, `len(HQ_STATE_CODE) == 2`, `len(HQ_POSTAL_CODE) == 5 and != ''` for every row. Symmetric in shape with Plan 8's NULL-semantic invariants but lighter-weight (no enum-driven gating).
  - **General rule per spec v1.5:** assume any string column from V_ACCOUNT_ANCHORS may carry empty strings, dirty values, or unexpected widths. Defensive SQL/Python beats post-deploy fix-up.
- **Two NULLable columns gated by data-availability heuristics, NOT enums.** This is meaningfully simpler than Plan 8's enum-driven NULL gating because the BUSINESS audience is large enough (12,021 rows) for distributional rate convergence to work, unlike Plan 8's narrow 3,920-row Wealth cohort where per-anchor invariants were needed:
  - `WEBSITE_DOMAIN` is NULL when normalized ACCOUNT_NAME alnum-slug length < 3 (~0.5% of rows — almost all rows have it).
  - `TECH_STACK_FLAGS` is NULL when the industry-biased tag count rolls 0 (~10% of rows — concentrated in Personal Services / Construction / F&B).
  - The L1 tests assert these distributionally (boring-case rates: ~99.5% non-null and ~90% non-null) over a 3+ month roll (~30 rows per anchor cohort), rather than per-anchor — works at 12K-row scale where Plan 8 needed per-anchor.
- A given account's TECH_STACK_FLAGS / FOUNDED_YEAR draw / LINKEDIN_FOLLOWERS may flip month-over-month with the seed roll — that's intentional. EMPLOYEE_BAND, REVENUE_BAND, NAICS, SIC, WEBSITE_DOMAIN, and HQ geography are all deterministic from the anchor and **must be byte-identical month-over-month** (per L1 per-anchor consistency invariants).
- **Anchor reads.** `ACCOUNT_NAME`, `INDUSTRY`, `ANNUAL_REVENUE`, `EMPLOYEE_COUNT`, `COUNTRY_CODE`, `STATE_CODE`, `POSTAL_CODE`. INDUSTRY / ANNUAL_REVENUE / EMPLOYEE_COUNT all populated for the BUSINESS cohort; the three string columns require defensive projection (above). When EMPLOYEE_COUNT is NULL/0, default `EMPLOYEE_BAND='1-10'`. When ANNUAL_REVENUE is NULL/0, default `REVENUE_BAND='<$1M'`.
- The DLO → DMO field mapping must be completed in DC Setup UI for fully-custom DMOs (the API endpoint returns 500). See the recipe at `../Snowflake_Claritas_Demographics/docs/dc-setup-recipe.md` from Plan 1 T7.
- Snowflake DATE columns auto-discover as `MM/dd/yyyy` in the DC data-stream POST body. Use that format for `PROFILE_MONTH` and `LAST_DATA_REFRESH_DATE`'s `sourceFields` entries, NOT `yyyy-MM-dd`.
- DC mapping descriptions must be ≤521 chars (Plan 6 finding). Trim DMO descriptions if drafting against the rowspec verbatim.
- 14 columns total: 12 NOT NULL + 2 NULLable (`WEBSITE_DOMAIN`, `TECH_STACK_FLAGS`).
- `write_pandas(auto_create_table=True)` mis-types `datetime64[ns]` as `NUMBER(38,0)`. The MERGE source SELECT casts back via `TO_TIMESTAMP_NTZ(GENERATED_AT::NUMBER / 1000000000)` — see template Task 4 §_merge.
- The MERGE handles 2 NULLable VARCHAR columns (`WEBSITE_DOMAIN`, `TECH_STACK_FLAGS`) — `None` passes through `write_pandas` → MERGE without special-casing.
- Salt is `"zoominfo"` (monthly cadence, single-salt, in the SP module constant only). No year-stable subfield (vendor case ID, DUNS, etc. — none of those exist in this schema).
- DC PK collapses to single-column `profileMonth__c` with `ssot__AccountId__c` as a KQ qualifier (single-column-PK rule from Plan 4).
- NAICS / SIC declared as VARCHAR (string codes, not numeric — preserves leading zeros and matches the NAICS/SIC publication format).
- Cadence is **monthly** (`'USING CRON 0 7 1 * * UTC'` — first of the month at 07:00 UTC). Matches Plans 1-3, 6, 8.
- **Volume.** ~12,021 rows/month. Re-runs same month MERGE-replace in place; live storage grows by ~12,021 rows per month.
