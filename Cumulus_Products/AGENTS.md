# AGENTS.md — Cumulus_Products

Context for AI coding agents working on the **Cumulus Bank product brochure catalog** — 55 multi-page PDFs spanning Retail, Wealth, and Commercial banking, generated from a shared ReportLab brand system with segment-aware theming.

For full catalog structure, brand-system overview, and rebuild instructions, see [README.md](README.md). This file is the agent-orientation primer.

# Product context

This is a **demo asset**, not a Salesforce DX project. Despite living in the JDO monorepo (which is heavily DX-oriented), Cumulus_Products is Python-only:

- 55 finished PDFs in `brochures/` (one per product)
- 55 generator scripts in `generator/build_*.py` (one per product)
- 1 shared brand module `generator/brand.py` (the design system: themes, flowables, charts)
- Canonical product specs in `docs/PRODUCT_SPECS.md` (rates, fees, terms, voice guide)

> **Cumulus Bank is a fictitious institution.** All rates, fees, disclosures, and product terms are illustrative only — never use these materials for consumer marketing, regulatory filings, or any actual financial-decision basis.

# Tech stack

- **Python 3.11+**
- **ReportLab** — PDF generation (the entire layout engine)
- **matplotlib** — charts (growth curves, amortization, donuts) embedded as PNG into the PDFs
- **PyMuPDF** — QA / inspection (text extraction, page-count verification)
- **No `force-app/`, no Apex, no LWC, no DX project file.** This breaks the "each JDO top-level folder is a DX package" convention deliberately — it's a content asset, not a Salesforce deployable.

# Commands

```bash
# Set up the build environment (one-time)
python3 -m venv .venv && source .venv/bin/activate
pip install reportlab matplotlib pymupdf

# Rebuild a single brochure
cd generator
python3 build_premier_checking.py
# → writes ../brochures/01_Personal_Deposits/Cumulus_Premier_Checking.pdf

# Rebuild every brochure (sequential)
for f in build_*.py; do python3 "$f"; done
```

There's no `requirements.txt` checked in — versions float to whatever pip resolves. Pin if a regression appears after a major ReportLab release.

# Architecture

Every brochure is generated by the same recipe:

```python
import brand as B
B.set_theme("retail" | "wealth" | "commercial")          # picks accent + voice
doc = B.BrochureDoc(out_path, product_name=..., product_code=...,
                    category=..., segment=...)
story = []
story += B.hero_block(product_name=..., lede=..., summary_rows=[...])
story += B.switch_to_body()
story.append(B.section_header("Overview", kicker="At a glance"))
story.append(B.lead_para("..."))
# ... more sections, tables, charts ...
story += B.disclosure_block("Important disclosures", B.STANDARD_DEPOSIT_DISCLOSURES + [...])
story += B.back_cover_block()
doc.build(story)
```

The shared `brand.py` module owns:

- **Three segment themes** (Retail / Wealth / Commercial) — share one structural skeleton, differ only in accent color and editorial voice. See [README.md § Brand system](README.md#brand-system).
- **All flowables** (hero, section header, lead para, feature grid, callout, disclosure block, back cover) — every script composes from this vocabulary, never builds custom layout primitives.
- **Standard disclosure constants** — `STANDARD_DEPOSIT_DISCLOSURES`, `STANDARD_LENDING_DISCLOSURES`, etc. Product scripts append their product-specific disclosures.

# Conventions

## Brand discipline

- **`brand.py` is the contract.** If you need a new layout primitive (a new flowable, a new chart helper), add it to `brand.py`. Don't inline it in a single `build_*.py` — that fragments the design system.
- **Three segment themes only.** Don't introduce a fourth segment without a product-level decision; the navy / teal / champagne / copper palette is part of the demo's visual identity.
- **`product_code`, `category`, `segment` arguments are required** on `BrochureDoc()` because the footer references them on every page. Missing these produces a brochure that fails the visual QA pass.

## Per-product script discipline

- **One `build_<name>.py` per product.** No multi-product scripts.
- **The script's job is content, not layout.** Layout primitives come from `brand.py`. Charts come from `brand.py` chart helpers. The script provides the words, numbers, and section ordering.
- **Section order matches [README.md § Brochure structure](README.md#brochure-structure-per-product).** Don't reorder per-script — consistency across the catalog is part of the brand.
- **Numbers come from `docs/PRODUCT_SPECS.md`.** That's the canonical source of rates, fees, and terms. Don't drift per-script.

## Disclaimer integrity

Every PDF must include the "fictitious institution / illustrative only" disclaimer. Removing or weakening it is **not a stylistic choice** — it's a guard against the demo being mistaken for an actual product. The standard disclosure constants in `brand.py` already include the framing; if you write a new disclosure block, preserve the framing.

# Common mistakes

- **Inlining layout primitives in `build_*.py`** — fragments the design system. Add to `brand.py` instead.
- **Drifting rates / fees / terms from `docs/PRODUCT_SPECS.md`** — the spec is the source of truth. Per-script edits create silent inconsistency across the catalog.
- **Adding a new product without choosing a segment** — `BrochureDoc()` requires it, and the footer + back cover renderer use it for theming. Default to Retail only when the product is genuinely retail-tier.
- **Removing the fictitious-institution disclaimer** — never. The catalog could be mistaken for an actual product if this slips.
- **Treating this as a DX package.** It's not. There's no `sfdx-project.json`, no `force-app/`, no `sf` deploy. PRs touching deploy plumbing should leave this folder alone.
